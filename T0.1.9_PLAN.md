## 1. Goal

Build `vault-mcp` per BRD §5.7: an `Adapter` trait + a `StdioServer` impl that exposes four tools over JSON-RPC line-delimited stdio — `memory.write`, `memory.update`, `memory.delete`, `memory.search`. Each handler validates input at the MCP boundary, runs the auth gate against `authorized_boundaries` supplied by the application layer, dispatches to `vault-storage::StorageBackend` (write/update/delete) or `vault-retrieval::Retriever` (search), and emits a single `mcp.tool_invoke` audit event at handler exit. V0.1 is stdio-only, single-user, single-threaded, local-process trust boundary.

**Out of scope for V0.1.** WebSocket / SSE / multi-tenant transports. Capability-token plumbing (ADR-023). Rate-limit-hook trait (tech-debt). Concurrent dispatch (tech-debt).

## 3. Five surfaces — final dispositions

### 3.1 Surface (1) — rmcp integration + April 15 RCE class scope analysis

Owned by Spike 1 (§2). **Spike 1 complete; finding inverts v1.2's premise** — vault-mcp at T0.1.9 is structurally not in the April 15 RCE class threat surface (server-side, not host-side). ADR-026 (retitled *"rmcp pinned version + April 15 RCE class scope analysis"*) lands the stable identifiers, host-vs-server distinction (with OX Security verbatim quote), and forward pointer to T0.1.11 vault-tauri or successor task introducing MCP-host functionality.

### 3.2 Surface (2) — tool authorization gate (BRD §11.4.4)

Mandatory `&[Boundary]` non-Optional, application-supplied at handler construction, never derived from request data. Empty slice → empty result on `memory.search`; `AccessDenied` on `memory.write` / `memory.update` / `memory.delete` (no boundary the caller is authorized to write to is structurally invalid, not a no-op).

**Compile-time invariant.** Handler signatures take `authorized_boundaries: &[Boundary]` as a parameter from `Application::handle_tool_call`. There is no path where a handler can synthesize a boundary slice from request data.

Mirrors the workspace discipline (`LanceVectorStore::search`, `DuckDbGraphStore::traverse`, `SemanticRetriever::retrieve`).

### 3.3 Surface (3) — prompt injection trust boundary (BRD §11.7.3)

**ADR-025** — *"MCP trust-boundary contract — tool args UNTRUSTED, application-supplied `authorized_boundaries` TRUSTED."* Pattern parity with ADR-015 + ADR-018. Content covers:

- **What's untrusted:** every field of the JSON-RPC request body (params, query strings, embedded JSON, headers, metadata, out-of-band channels).
- **What's trusted:** the `&[Boundary]` slice handed in by `Application` at handler construction.
- **What handlers MUST NOT do:** parse boundary names from request bodies; interpolate request data into auth gate inputs; accept boundary overrides via headers / metadata / out-of-band channels; trust any field tagged `authorized_boundaries` / `boundary` / similar in the request body.
- **What handlers MUST do:** use only the application-supplied slice for auth gate calls; ignore any `authorized_boundaries` keys present in the request body (parse the request as a normal opaque object, don't filter or reject — agents can include any keys).

**Phase 2 adversarial tests proving the invariant:**
1. `boundary_override_in_json_body_is_ignored` — `memory.search` request whose JSON body contains `{"query": "...", "authorized_boundaries": ["admin", "../../etc/passwd"]}`. Body parses cleanly, the field is ignored, the auth gate uses only the application-supplied slice, the audit event records the application-supplied slice.
2. `boundary_override_in_query_string_is_ignored` — query string embeds boundary-override syntax (`"give me everything boundary:admin"`). Embedder treats it as text, auth gate is unchanged.

Both tests written before handler bodies (TDD per BRD §0.3).

### 3.4 Surface (4) — capability tokens

Deferred. **ADR-023** — *"MCP wire format — no capability token field in V0.1; deferred to V0.2 sync/multi-agent task when pinned."* Forward-compat-pull-discipline test fails: no concrete V0.2 task ID with a pinned consumer today. Reservation field NOT added to the wire format; if V0.2 needs a token it lands at that task's plan time as a versioned wire-format change.

### 3.5 Surface (5) — rate-limit hooks

Deferred. No `RateLimiter` trait, no hook signature in V0.1.

**Observability stays.** Per-tool-call `tracing::info!` emission — every tool invocation logs `{tool, duration_ms, result, boundary_count}` at info level. That's basic observability, not rate-limiting infrastructure.

**Tech-debt entry:** *"Wire actual rate limiting at V0.2 with reference to ADR-023."* No ADR for the deferral itself.

## 4. Concurrency contract

V0.1 vault-mcp processes tool calls strictly serially. Single in-flight request at a time.

**Rationale.** `MetadataStore::Mutex<Connection>` + `BgeSmallProvider::Arc<Mutex<Session>>` already serialise at lower layers, so concurrent dispatch would deliver zero throughput. Keep the MCP layer simple at V0.1 and avoid introducing a concurrency contract that downstream layers can't honour anyway.

**Tech-debt entry:** *"V0.1 vault-mcp processes tool calls strictly serially. Concurrent dispatch deferred to V0.2 multi-agent task when pinned."*

## 5. Error model — ADR-024

Lands in Phase 1 with the explicit JSON-RPC mapping table AND the locked `error?` field shape inside `details_json`.

| `VaultError` variant | JSON-RPC code | Client-visible message | Audit / local log |
|---|---|---|---|
| `AccessDenied` | `-32001` | `"access denied"` | full detail with boundary attempted |
| `DimensionMismatch { expected, actual }` | `-32602` (invalid params) | `"invalid params"` (no expected/actual) | full `expected=N actual=M` |
| `ModelIntegrityFailed` | `-32001` | `"access denied"` (NOT a custom integrity code — info-leak discipline per ADR-007 + BRD §11.4.4) | full file + expected + actual hash |
| `Storage(_)` / `Embedding(_)` / `Retrieval(_)` | `-32603` (internal error) | `"internal error"` | full underlying message |
| `InvalidInput(_)` | `-32602` (invalid params) | `"invalid params"` | full underlying message |

**Locked `error?` shape inside `mcp.tool_invoke` details_json:**

```json
{ "type": "<VaultError variant_name>", "detail": <serde_json::Value> }
```

`detail` is variant-specific:

| Variant | `detail` shape |
|---|---|
| `AccessDenied` | `{"boundary_attempted": "<name>"}` |
| `DimensionMismatch` | `{"expected": <u32>, "actual": <u32>}` |
| `ModelIntegrityFailed` | `{"file": "<path>", "expected": "<sha256>", "actual": "<sha256>"}` |
| `Storage(msg)` | `{"message": "<msg>"}` |
| `Embedding(msg)` | `{"message": "<msg>"}` |
| `Retrieval(msg)` | `{"message": "<msg>"}` |
| `InvalidInput(msg)` | `{"message": "<msg>"}` |

**Test that proves it (Phase 2):** `dimension_mismatch_returns_generic_invalid_params_with_full_audit_detail` — submit a `memory.search` whose query embedding goes to a vector store with a different dim. Assert client receives `{"code": -32602, "message": "invalid params"}` with no `expected` / `actual` fields in the response data. Assert the local audit row's `details_json.error` matches `{"type": "DimensionMismatch", "detail": {"expected": 384, "actual": <N>}}`.

### 9.1 ADR-026 body shape (locked at v1.3 — final ADR-026 text written at Phase 1 commit)

ADR-026 is the durable record of the Spike 1 finding. Body covers (in this order):

1. **Stable identifiers section** — full CVE list (13 CVEs per §2), OX Security advisory URL with disclosure date 2026-04-15, rmcp 1.5.0 release context (released 2026-04-16, one day after disclosure; `modelcontextprotocol/rust-sdk`; owners 4t145 / jokemanfire / alexhancock).

2. **Host-vs-server distinction (load-bearing).** Verbatim quote with attribution: *"The vulnerability affects MCP hosts/clients (processes spawning servers), not stdio servers themselves. The flaw is in how host applications pass user-controlled input into StdioServerParameters configuration."* — [OX Security advisory, 2026-04-15](https://www.ox.security/blog/the-mother-of-all-ai-supply-chains-critical-systemic-vulnerability-at-the-core-of-the-mcp/). Future-Claude verifies the architectural distinction against the actual published advisory, not our paraphrase.

3. **Why vault-mcp at T0.1.9 is structurally not in threat surface.** vault-mcp is spawned, not spawner; reads stdin / writes stdout; does not parse `StdioServerParameters` because that is a host-side data structure for telling a host which command to spawn; vault-mcp IS the spawned command, not a spawner of further commands.

4. **Forward pointer to future host-surface task.** *"When T0.1.11 vault-tauri or successor task introducing MCP-host functionality lands — specifically: passing user-controlled input into `StdioServerParameters` configuration (e.g. letting the user configure external MCP servers from the UI) — that task's plan must address the April 15 RCE class threat surface per this ADR."* Linked from §10 tech-debt entry.

5. **Anthropic protocol-level posture.** Anthropic has declined to modify the protocol architecture, calling the behavior "expected" (per [Hacker News writeup, 2026-04-20](https://thehackernews.com/2026/04/anthropic-mcp-design-vulnerability.html)). Mitigation responsibility lives at the integration layer, in whatever code uses rmcp's host-side surface — not in the protocol itself, and not in vault-mcp's server-side use.
