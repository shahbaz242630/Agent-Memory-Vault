## Five pre-design questions (answer before any spikes or code)

### Q1 — Model file distribution: bundle, FP32, location-by-caller

**Decision.** **Bundle the bge-small-en-v1.5 ONNX model + tokenizer files in the vault distribution.** No first-run download. **Use the FP32 variant** (not INT8 quantized) for V0.1.

**Refinement on opening position — file location is a layered concern:**
- **In the `vault-embedding` crate (T0.1.7):** the crate accepts a `&Path` to a model directory and loads from it. No assumption about where that path resolves to. Tests use `crates/vault-embedding/test-fixtures/bge-small-en-v1.5/` (gitignored — model files are not committed; a dev-setup script downloads them once per checkout). This keeps the crate decoupled from production-bundle mechanics.
- **In `vault-app` (T0.1.10):** wires the path resolution — looks first at a config-overridable path, falls back to a known location relative to the binary. This is where production path resolution logic lives.
- **In `vault-tauri` (T0.1.11):** the model files ship as Tauri **resource bundles** (Tauri's `tauri.conf.json` `resources` array). Path is resolved via `tauri::PathResolver`. This is the actual end-user distribution surface.

**FP32 vs INT8 choice (refinement on opening position).** bge-small-en-v1.5 ships in two ONNX flavours: FP32 (~130 MB) and INT8 quantized (~33 MB). For V0.1 founder-only alpha, **bundle FP32** because:
- Quantization introduces a quality risk (small but measurable cosine-similarity drift) that we don't want stacked with all the other V0.1 unknowns.
- The 100MB delta is negligible against the eventual Phi-4-mini bundle (~2.5GB at V0.2.x).
- INT8 inference is ~2× faster on CPU, but bge-small FP32 inference on M1 is already <50ms per single embedding — well under BRD §5.3's <100ms target. No urgency.
- Revisit at V0.2.0 / V1.0 if binary-size pressure or batch-throughput pressure makes the quantization tradeoff worth re-evaluating. Until then, FP32.

**Reasoning (overall).** Bundling means model integrity is verified at install time (signed installer chain, BRD §11.7.5) rather than depending on runtime network verification. No first-run network dependency aligns with "local-first, zero-knowledge" (BRD §1). Founder dogfood works offline. Simpler control flow — no first-run download UI, no checksum-verify-then-retry path, no network-failure error class.

**Alternatives considered.**
- *First-run download with checksum verify:* rejected. Adds network failure surface, requires UI for download progress + retry, blocks first-run UX. The size savings (130MB shaved off installer) doesn't justify this for founder dogfood; revisit at V1.0 if installer size becomes a customer-friction point.
- *Bundle INT8 quantized:* rejected for V0.1 (quality risk + no urgent perf need). Reconsider at V0.2 if mobile distribution materialises.
- *Ship a "downloader stub" that pulls the model on first run from a CDN we control:* rejected. Adds CDN ops + monitoring, plus the network-dependency problem above.

**ADR candidate:** none required for Q1 alone — bundling is the obvious default given the BRD's local-first posture. The location-resolution layering will be re-stated in HANDOFF.md as a locked-into-code invariant once T0.1.10 + T0.1.11 implement the production paths.

### Q2 — ONNX Runtime native library: bundle dylib via `load-dynamic`

**Decision.** **Use `ort 2.0`'s `load-dynamic` feature.** Bundle the `libonnxruntime` dylib per platform (`.dylib` on macOS, `.dll` on Windows, `.so` on Linux) alongside the app binary via Tauri's resource bundling at T0.1.11. The vault-embedding crate accepts a `lib_path: &Path` at provider construction; vault-app resolves the production path; tests point at a downloaded ORT lib in `target/test-fixtures/onnxruntime/`.

**Why `load-dynamic` (the load-at-runtime feature flag) and not the default `download-binaries`:**
- **Deterministic builds.** `download-binaries` (ort's default in 2.x) downloads the ORT prebuilt at *build time* from Microsoft's CDN. CI / clean-checkout builds therefore depend on Microsoft's CDN being up. `load-dynamic` lets us check the dylib into our own distribution path, fully under our control.
- **Single-source-of-truth for the dylib version.** With `load-dynamic`, the dylib version is pinned by the file we bundle, not by ort's build-time fetch. CVE patching is a "swap the file" operation, not a rebuild.
- **Cleaner signed-binary story (BRD §11.7.5).** Our installer signs the dylib alongside the app binary. With `download-binaries`, the dylib is fetched and stored in cargo's target dir — harder to integrate into the signed-distribution chain.

**Path-resolution ergonomics (matches Q1's layering):**
- T0.1.7 (vault-embedding crate): provider takes `lib_path: &Path` + `model_path: &Path` at construction.
- T0.1.10 (vault-app): `Application::new` resolves both paths.
- T0.1.11 (vault-tauri): `tauri::PathResolver` finds the bundled paths in the production package.
- Tests / dev: a `scripts/setup-dev-env.sh` (Mac/Linux) + `.ps1` (Windows) helper downloads the ORT prebuilt into `target/test-fixtures/onnxruntime/` once per checkout. Tests skip with a clear "run setup script" message if not present.

**Static link rejected** because: (a) ORT static builds are fragile per platform (Microsoft doesn't ship them officially), (b) static-link binary size is larger than dynamic-link + shipped dylib, (c) static-link CVE patching means rebuild + redistribute the entire app vs swap a single file.

**System-installed-ORT rejected** because: (a) founder has to manually install ORT for V0.1 alpha, (b) any future user has to do the same, (c) version-skew across user machines becomes a support nightmare.

**Spike candidate:** ort 2.0 API surface + `load-dynamic` setup mechanics — see spike list below. This is the highest-risk-of-API-drift area of T0.1.7 (ort had significant churn between 1.x and 2.x), so the spike runs before the implementation phase locks in.

**ADR candidate:** **ADR-019 — ort native-lib distribution: load-dynamic + bundled dylib + path-resolution layering.** Documents the choice and the per-platform dylib bundling pattern.

### Q3 — Embedding cache: not in V0.1

**Decision.** **No embedding cache in V0.1.** `BgeSmallProvider::embed(text)` is stateless — every call runs fresh inference.

**Reasoning.**
- V0.1 callers don't re-embed the same text often:
  - **Write-path (`StorageBackend::write_memory`):** one embed per memory write. Each memory has unique text.
  - **Search-path (T0.1.8 vault-retrieval):** one embed per search query. Repeated identical queries within one session are rare in real founder usage.
- Inference is ~5–50ms on M1 (varies with input length). Cache hit savings are negligible at V0.1's expected throughput (handfuls per minute).
- A cache adds: LRU eviction policy, max-size config, cache-invalidation-on-model-change logic, an extra source of state to test. None of this complexity earns its keep at V0.1's call patterns.

**Revisit triggers:**
- T0.2.x consolidator (BRD §5.6) when re-embedding many memories during a sleep cycle — but the consolidator can build its own batch-with-internal-dedup logic if needed.
- V0.2 sync — if cross-device sync ends up re-embedding the same memory after pull from a peer, evaluate then. (Likely not — the embedding is part of the synced payload.)
- V1.0 connectors — if Gmail/Calendar ingestion produces highly repetitive snippets that re-embed identical strings.

**Alternatives considered.**
- *Simple hash-keyed LRU on the `embed` call:* rejected for V0.1 per "Reasoning" above.
- *Disk-backed embedding cache (persisted in SQLite alongside memory rows):* rejected as feature creep — that's effectively storing the embedding in SQLite, which we already do at T0.1.4 indirectly via LanceDB. Different concern.

### Q4 — Batch embedding API: V0.1 ships `embed` only; `embed_batch` deferred

**Push-back on opening position.** Define **`embed(text: &str) -> Vec<f32>` only** in V0.1. `embed_batch(texts: &[&str]) -> Vec<Vec<f32>>` is **deferred** until a real batching caller exists.

**Why push back:**
- **CLAUDE.md hard rule:** "Don't add features beyond what the task requires. ... Don't design for hypothetical future requirements." Building `embed_batch` for the consolidator (T0.2.x) is exactly the speculative scaffolding the rule forbids. Same reasoning that deferred `FullySynced` in T0.1.6 (ADR-018 Item A).
- **No V0.1 caller batches.** Write-path is one memory at a time; search-path is one query at a time. No exercising caller, no design feedback, easy to get the batch API shape wrong in the absence of a consumer.
- **Trait-additive when needed.** Adding `embed_batch` as a default-impl trait method later (with an iteration-over-`embed` default, then a real batched override on `BgeSmallProvider`) is non-breaking. The API can grow when the consolidator (T0.2.2) or a real batch-bound caller surfaces.
- **The ORT batching benefit is real but speculative for V0.1.** ORT can batch inference efficiently, but the throughput win matters at consolidator scale (re-embedding hundreds/thousands of memories in one sleep cycle), not at write-path scale (one at a time).

**If push-back is rejected:** define both. Trait shape would be:
```rust
#[async_trait]
pub trait EmbeddingProvider: Send + Sync {
    async fn embed(&self, text: &str) -> VaultResult<Vec<f32>>;
    async fn embed_batch(&self, texts: &[&str]) -> VaultResult<Vec<Vec<f32>>> {
        // default: iterate embed; impls can override for ORT-batched
        let mut out = Vec::with_capacity(texts.len());
        for t in texts { out.push(self.embed(t).await?); }
        Ok(out)
    }
}
```
This would still respect the "no premature impl" instinct — the trait *reserves* `embed_batch` but `BgeSmallProvider` only overrides it when batch perf becomes the bottleneck.

**Recommended:** push-back stands — V0.1 trait has `embed` only. Keep the API surface small until the consolidator forces the question. The cost of adding `embed_batch` later is minimal (additive trait method); the cost of getting its shape wrong now and rewriting downstream callers is non-zero.

### Q5 — Model + tokenizer integrity: compiled-in SHA-256, paired-files invariant

**Decision.** **At provider construction, hash the model file AND the tokenizer file with SHA-256 and compare against constants compiled into the binary.** Mismatch on either → fail with `VaultError::ModelIntegrityFailed { file: String, expected: String, actual: String }`. Both files must be paired — a model from version X with a tokenizer from version Y silently produces shifted embeddings, which is the exact failure class this check exists to prevent.

**Refinement on opening position:**
- **Hash both files, not just the model.** The opening position said "model file similarly checked" for the tokenizer — making it explicit: the tokenizer config (`tokenizer.json`) is a separately-versioned artefact, and a mismatched tokenizer with a matched model is the most insidious failure (no error, just silently bad embeddings). Both hashes are compiled-in; both checks fire on every load.
- **Hash the *file bytes*, not parsed content.** Defends against any whitespace/format normalization on disk that might happen during install (e.g., line-ending conversion on Windows). The bytes-on-disk are the source of truth; `tokenizers` and `ort` consume bytes.
- **Constants live in `vault-embedding/src/integrity.rs`** as named consts: `BGE_SMALL_EN_V1_5_MODEL_SHA256` and `BGE_SMALL_EN_V1_5_TOKENIZER_SHA256`. Updates to these consts require a code change + commit + review — operationally hard to bypass.
- **Test: integrity check fails with a clear error** when either file is mutated by a single byte. Property test fuzzes random byte positions in a copy of the file; every mutation fails the check.

**Reasoning.** This is the BRD §11.6 control point. Without it, a future bug (or supply-chain attack on the model distribution) silently produces wrong embeddings, retrieval gets subtly worse, and we never know. Compiled-in hashes mean the verification is part of the signed-binary chain (BRD §11.7.5) — an attacker has to compromise both the binary signing and the file distribution to swap the model undetected.

**Application-level failure mode (locked here so T0.1.10 / T0.1.11 inherit a clear contract).** Integrity-check failure is **fatal at startup, no fallback, no degraded mode.** Concretely:

1. `BgeSmallProvider::open(...)` returns `Err(VaultError::ModelIntegrityFailed { .. })`.
2. `Application::new` (T0.1.10) propagates the error — `Application::new` returns `Err`, the application never finishes constructing.
3. `vault-tauri` (T0.1.11) catches the error at `tauri::Builder::run` time, **shows a fatal-error dialog** ("Memory Vault cannot start: model integrity check failed. Reinstall the application to recover. Details: <hash mismatch info>"), and **exits with non-zero status**. No partial UI, no read-only mode, no "continue without embeddings" path.

**Why fatal-fail and not degraded mode:** Embedding is **mandatory** for the cascading orchestrator's write-path — `StorageBackend::write_memory(&self, memory: &Memory, embedding: &[f32])` takes the embedding as a non-optional parameter (T0.1.6 lockstep contract). A degraded-mode "memories written without embeddings" path would mean either (a) memories enter SQLite but never reach LanceDB (the exact divergence problem T0.1.6 just shipped a detector for), or (b) we'd have to invent a synthetic-zero-embedding sentinel that LanceVectorStore would happily store but cosine-search would return as garbage. Both are strictly worse than fail-fast at startup. Reinstall is a clear, recoverable user action; silent embedding corruption is not.

**Audit trail:** the integrity-failure path emits a CRITICAL audit event (`store.integrity_failure` with `model_path`, `expected_sha`, `actual_sha`) BEFORE the fatal exit, written to the metadata store via the same `tx_append_audit` path used elsewhere. If SQLite is also unreachable at this point (very early in startup), fall back to `tracing::error!` so the operator at least sees the diagnostic. T0.1.10 wires this; T0.1.7 ensures the error type carries enough info for T0.1.10 to populate the audit event without re-hashing.

**Why SHA-256 not BLAKE3:** the rest of the codebase uses BLAKE3 (audit chain, AAD), so consistency would argue BLAKE3. But: SHA-256 is universally available, the file hashes will appear in BRD §5.3 documentation + model-card references + any future security audit, and reviewer-facing hash strings are more familiar in SHA-256. The cost of "two hash algorithms in the workspace" is minimal; the comprehension cost of explaining "why BLAKE3 for these files" in a security review is non-zero. **SHA-256 here.**

**ADR candidate:** **ADR-020 — Model + tokenizer integrity scheme.** Documents the paired-files invariant + the compiled-in-SHA-256 mechanism + the failure-mode test discipline + the "always verify both, not just the model" rule for any future model crate (Phi-4-mini at T0.2.1+).

## Module breakdown (vault-embedding crate)

```
crates/vault-embedding/

  src/
    lib.rs                            # crate root + #![forbid(unsafe_code)] + #[allow(unsafe_code)] on the ort module only

    provider.rs                       # EmbeddingProvider trait (embed only; no embed_batch in V0.1)
    bge_small.rs                      # BgeSmallProvider impl: wraps ort::Session + tokenizers::Tokenizer
    integrity.rs                      # SHA-256 verification + compiled-in expected hashes
    inference.rs                      # the spawn_blocking wrapper around session.run + L2-normalize helper
    ort_module/                       # ALL ort FFI lives here; #[allow(unsafe_code)] scoped to this module per ADR-002
      mod.rs                          # session-loading via load-dynamic; environment singleton
  tests/                              # integration tests (medium per BRD §7.1)
    end_to_end.rs                     # known-text → known-shape, L2-normalized, deterministic
    integrity_failure.rs              # mutated model / tokenizer → ModelIntegrityFailed
    cosine_sanity.rs                  # similar texts cosine-close, dissimilar texts cosine-far
  test-fixtures/                      # gitignored — populated by scripts/setup-dev-env helper
    bge-small-en-v1.5/
      model.onnx
      tokenizer.json
```

**File-size discipline:** all source files target ≤ 300 lines; the BRD §2.5 cap is 500 lines, but per the project memory "500-line cap is soft" — split on tangled-responsibility signals. The ort_module is the only place that touches FFI; isolation simplifies the ADR-002 unsafe-carve-out review.

**Workspace deps to add (exact-pinned per BRD §2.9 / T0.1.5b after first build):**
- `ort` — currently caret-pinned `"2.0"` in workspace `Cargo.toml`. **MUST exact-pin to a specific 2.0.x version** (e.g. `"=2.0.0-rc.10"` or whatever stable line resolves) from `Cargo.lock` after first build. Default features audited via Spike 4 — if CUDA / TensorRT / DirectML are pulled in by default, switch to `default-features = false` + minimal feature set.
- `tokenizers` — currently caret-pinned `"0.20"`. Exact-pin to specific 0.20.x after first build.
- `ndarray = "0.15"` (likely required for tensor shape manipulation; confirm during Spike 1). Add only if needed; exact-pin if added.
- `sha2 = "0.10"` for SHA-256 (lightweight; no transitive surprises expected). Exact-pin.

**Workspace pinning discipline (load-bearing — flagged here because ORT is the highest-churn dep we've added since T0.1.4 LanceDB).** Per BRD §2.9 + the T0.1.5b sweep + ADR-013's chrono pin pattern: every dep with a workspace-member consumer is exact-pinned (`"=X.Y.Z"`, not `"^X.Y"`). For ORT specifically, the 1.x → 2.x transition reorganised session creation, input/output handling, and execution-provider config — a future `cargo update` that floats us from `"2.0"` to `"2.1"` could break the entire embedding crate silently. Exact-pin defends against that. **Operational follow-up:** add a recurring monthly ORT release-notes check (alongside the existing OpenSSL / protobuf / chrono CVE checks per ADR-006 / ADR-011 / ADR-013) — first Monday of each month, scan https://github.com/microsoft/onnxruntime/releases for security advisories or breaking changes affecting the pinned version. Tracked in HANDOFF.md tech-debt at T0.1.7 commit time.

## Test strategy (Medium per BRD §7.1)

Seven specific tests:

1. **Single embedding shape + dimension.** Embed `"hello world"`. Assert returned `Vec<f32>` has length **384** (matches `LanceVectorStore::dimension()` from T0.1.4).
2. **L2-normalization invariant.** Same input. Assert `embedding.iter().map(|x| x*x).sum::<f32>().sqrt()` is within `1e-5` of `1.0`. Load-bearing — `LanceVectorStore` cosine-distance is calibrated for L2-normalised inputs (T0.1.4 invariant).
3. **Determinism.** Two calls to `embed(same_text)` → byte-identical output. ONNX Runtime is deterministic given same inputs + same session; this test catches accidental nondeterminism (e.g., dropout layer left active, RNG seed drift).
4. **Cosine-similarity sanity.** Embed three texts: `"the cat sat on the mat"`, `"a feline rested on a rug"`, `"quantum chromodynamics"`. Assert `cosine(a, b) > 0.6` (semantically similar) AND `cosine(a, c) < 0.4` (semantically distant). Loose bounds — this is a sanity check, not a quality benchmark.
5. **Model integrity check.** Copy the bundled model to a tempdir, mutate one byte at a known offset, point provider at the mutated copy. Assert `BgeSmallProvider::open` returns `Err(VaultError::ModelIntegrityFailed { .. })`. Same test variant for the tokenizer file.
6. **Performance budget (BRD §5.3 acceptance).** Single `embed("a short sentence about cats")` completes in ≤ 100ms on M1 / x86_64-linux dev hardware. Marked `#[ignore]` by default so it doesn't fail in CI on slow runners; runnable explicitly via `cargo test -p vault-embedding -- --ignored embed_within_budget`.
7. **`spawn_blocking` correctness.** Invoke `embed` from inside an async test with a concurrent `tokio::time::sleep(50ms)`. Assert sleep completes in roughly 50ms (not blocked by inference time). Confirms the sync `ort::Session::run` is properly wrapped in `tokio::task::spawn_blocking` and doesn't starve the tokio reactor.
8. **`embed_output_is_l2_normalized` — broad-input invariant test (load-bearing).** Test 2 covers L2-normalisation for one fixed input; test 8 enforces the *contract* across diverse inputs. Embed **20 varied texts** (mix of: short single words, long paragraphs, punctuation-heavy, non-ASCII Unicode, repeated tokens, near-empty whitespace-stripped). For each output `v`, assert `(v.iter().map(|x| x*x).sum::<f32>().sqrt() - 1.0).abs() < 1e-6`. This is the test that catches a future "let's drop normalization for performance" change before it silently breaks `LanceVectorStore`'s cosine scoring (which is calibrated for unit-norm inputs per T0.1.4 invariant). The contract is "L2-normalized output across all inputs"; this test pins the contract.
9. **`embed_uses_cls_pooling_not_mean_pooling` — pooling-mode contract enforcement (load-bearing, added v1.2).** Per Spike 3 finding, bge-small uses CLS-token pooling (`1_Pooling/config.json`: `pooling_mode_cls_token: true`). Mean pooling produces vectors of correct shape and L2 norm but semantically wrong, and the existing tests (1, 2, 4, 8) would NOT catch a mean-vs-CLS regression because: shape is identical, normalisation is preserved by both, cosine-sanity weakens but doesn't fail randomly. **Test mechanics:** embed `"hello world"` via the production code path (CLS extraction + L2-normalise). Independently in test code, build a *mean-pooled* vector for the same input + same tokenizer + same session output (compute via the attention-mask-weighted average over `last_hidden_state[0, :, :]` then L2-normalise). Assert the two `Vec<f32>`s are NOT element-wise-equal within `1e-5` tolerance — confirms CLS extraction is in use. **Stronger version (Phase 4 integration smoke):** if a Python sentence-transformers reference output is available for `"hello world"` against bge-small-en-v1.5 (sentence-transformers honours `1_Pooling/config.json` by default, so its output IS the CLS-pooled-and-normalised reference), assert the production output matches the reference within `1e-4` tolerance per element. The contract is "CLS-token pooling per BAAI's 1_Pooling/config.json"; this test pins it explicitly.

**Test fixture provisioning.** `scripts/setup-dev-env.sh` (and `.ps1`) downloads the model + tokenizer from Hugging Face into `crates/vault-embedding/test-fixtures/bge-small-en-v1.5/` and verifies SHA-256. The script is idempotent — skips download if files exist with correct hashes. Tests panic with a clear "run scripts/setup-dev-env.sh first" message if the fixtures are missing.

## Implementation rhythm (after plan approval + spikes)

Mirror the T0.1.6 phased pattern. Each phase ends with all four DoD gates green per CLAUDE.md before commit-approval ask.

**Phase 0 — Spikes.** Run the spike-before-lock candidates (below) BEFORE phased implementation locks anything in. ~30–60 min total. Capture findings inline in this plan doc as a v2 amendment if anything material differs from the assumptions in Q1–Q5 / module breakdown.

**Phase 1 — Foundation.** Workspace dep additions (ort / tokenizers / sha2 / maybe ndarray) + crate skeleton + `EmbeddingProvider` trait + `VaultEmbeddingError` + `integrity.rs` with placeholder hash consts. Failing tests for the Medium-level test list above (now 9 tests including the new pooling-enforcement test). ~250 LoC. Single commit. **Spike 4 (ort default-features audit) folds in here** — run `cargo tree` after first build, classify ort's default features, trim to `default-features = false` + minimal explicit features if needed (likely just `load-dynamic` + `ndarray` + maybe `copy-dylibs`), land conditional ADR-021 in this commit if minimisation happened.

**Phase 1 implementation note (added v1.2 per Spike 1 finding).** `ort::init_from(dylib_path)` MUST be called exactly once before any other ort usage under the `load-dynamic` feature. Multi-threaded test runners (cargo test runs tests in parallel by default) mean each test thread might race to initialise. Wrap in `std::sync::Once::call_once(...)` inside a helper like `vault_embedding::ort_module::ensure_initialised(dylib_path: &Path)` — first call performs `init_from`, subsequent calls are no-ops. The helper is called at the top of `BgeSmallProvider::open` and is idempotent across the process lifetime. Tests construct `BgeSmallProvider` instances freely without worrying about init ordering.

**Phase 2 — Loading.** `BgeSmallProvider::open(model_path, tokenizer_path, ort_lib_path)` → integrity check → ORT session load → tokenizer load. Tests 1, 5 pass. ~200 LoC. Single commit.

**Phase 3 — Inference.** `embed(text)` end-to-end: tokenize → truncate → tensorize → `spawn_blocking(session.run)` → **CLS-token pool** (extract `last_hidden_state[0, 0, :]`) → L2-normalize. Tests 1, 2, 3, 4, 7, 8, 9 pass. Test 6 (perf) measured, recorded in commit message + HANDOFF.md. ~250 LoC. Single commit.

**Phase 3 implementation notes (added v1.2 per Spike findings):**
- **CLS-token pooling, not mean pooling (Spike 3 finding — load-bearing).** bge-small-en-v1.5's `1_Pooling/config.json` has `pooling_mode_cls_token: true`. The pooling step extracts `last_hidden_state[0, 0, :]` — the [CLS] token's vector at sequence position 0 — producing a `Vec<f32>` of length 384. Then L2-normalize. **Do NOT mean-pool.** Test 9 enforces this contract.
- **`u32 → i64` conversion at the tokenizer/ort boundary (Spike 2 finding).** `tokenizers::Encoding::get_ids() / get_attention_mask() / get_type_ids()` all return `&[u32]`. ONNX BERT models expect `int64` tensors. Convert per-element at tensor-construction time: `let ids_i64: Vec<i64> = encoding.get_ids().iter().map(|&x| x as i64).collect()`. Same for attention_mask and type_ids. Forgetting this surfaces as a loud ORT type-mismatch error (not silent), but the conversion is non-obvious from docs.rs alone — pinning here so future-Claude doesn't re-discover.
- **Manual `encoding.truncate(512, 0, TruncationDirection::Right)` after `encode` (Spike 2 finding).** The Rust `tokenizers` crate does NOT auto-apply truncation from `tokenizer_config.json`'s `model_max_length: 512` — that's Python tokenizers behaviour. For inputs longer than 512 tokens, manual truncation is required or ORT will reject the input with a position-embeddings-out-of-range error. For V0.1 single-input inference: truncate every encoding to ≤ 512 right after `encode`, no padding needed (single sequence; tensor shape is `[1, seq_len]` where `seq_len = encoding.len()` post-truncation).
- **Output extraction.** `session.run(...)?.get("last_hidden_state").unwrap().try_extract_array::<f32>()?` returns `ArrayViewD<f32>` with shape `[1, seq_len, 384]`. Slice to `[0, 0, ..]` for the CLS vector. The output name `"last_hidden_state"` matches BERT's standard ONNX export convention; verify via Spike 1's runtime confirmation (Phase 1 first build).

**Phase 4 — Integration smoke.** A lightweight integration test that wires `BgeSmallProvider` into a `StorageBackend::write_memory` call (against a tempdir vault), confirms the embedding makes it into LanceDB, and the dimension/normalization invariants hold end-to-end. NOT a full retrieval test (that's T0.1.8) — just confirms the embedding crate plugs in cleanly at the orchestrator's call site. ~100 LoC. Same commit as Phase 3 if small, otherwise its own.

**ADR-019 (ort native-lib bundling) + ADR-020 (model integrity scheme)** land in Phase 2's commit (the loading phase is where both decisions become load-bearing). Possibly **ADR-021 (ort feature minimization)** if the spike findings show default ort features pull in unwanted deps.

**(a) Artefact:** `crates/vault-embedding/examples/ort_api_spike.rs` (~80–120 lines). Long-term in-tree as executable documentation, mirroring `crates/vault-sync/examples/dryoc_spike.rs` (ADR-008) and `crates/vault-storage/examples/lance_corruption_spike.rs` (ADR-018). Also adds `ort = "=X.Y.Z"` to `crates/vault-embedding/Cargo.toml` `[dev-dependencies]` (NOT `[dependencies]` until Phase 1 adopts it).

**(b) Acceptance criteria (all must hold for spike PASS):**
1. `cargo run -p vault-embedding --example ort_api_spike` exits 0.
2. The spike loads the ORT dylib via `load-dynamic` from a path passed via env var or builder API (whichever ort 2.0 actually exposes — that's part of what the spike confirms).
3. The spike loads the bge-small-en-v1.5 ONNX model successfully via the actual session-creation API (`Session::builder().commit_from_file(...)` or whatever the 2.x equivalent is).
4. The spike runs inference on one fixed input (e.g., the token-id sequence for `"hello world"` after a hardcoded tokenizer call) and prints: output tensor shape, first 4 raw floats of the last-hidden-state tensor, and the L2 norm of the **CLS-pooled** embedding (i.e. `last_hidden_state[0, 0, :]` then L2-normalize — per Spike 3's `1_Pooling/config.json` finding, NOT mean-pooled).

6. Spike code documents (in comments) the *exact* type of `Session::run`'s output (`Vec<DynValue>`? typed `Vec<Tensor<f32>>`?) and the extraction pattern that worked.

**(c) On failure (any criterion not met):**

- Update the plan's Q2 + Module breakdown sections with the actual API shape — re-submit plan as v2 for re-review BEFORE Phase 1 starts.
- If `load-dynamic` doesn't work as expected (e.g., the feature is renamed, removed, or requires different setup), evaluate alternatives: (i) `download-binaries` default with build-time fetch, (ii) a different ort version line that does support load-dynamic, (iii) static link if ort offers it. Land an ADR if the choice deviates from Q2's decision.
- Outright spike failure (model doesn't load at all on any platform) = blocker; escalate to Shahbaz before any further T0.1.7 work.

### Spike 2 — `tokenizers` crate behaviour with bge-small *(Medium risk — silent-failure-prone)*

**Why:** padding / truncation / special-token handling for bge-small specifically matters for getting *correct* embeddings. Wrong handling produces output that's shaped right but semantically shifted — tests pass, retrieval quality silently degrades. Hardest class of bug to catch downstream.

**(a) Artefact:** `crates/vault-embedding/examples/tokenizer_spike.rs` (~60–80 lines). Independent of `ort_api_spike.rs` — separable concern, separate file. Adds `tokenizers = "=X.Y.Z"` to `[dev-dependencies]`.

**(b) Acceptance criteria (all must hold for spike PASS):**
1. `cargo run -p vault-embedding --example tokenizer_spike` exits 0.
2. `Tokenizer::from_file("test-fixtures/bge-small-en-v1.5/tokenizer.json")` loads without error.
3. For input `"the quick brown fox"`: spike prints token-id sequence, attention-mask sequence, and token-type-id sequence (if applicable). Token-id sequence has length consistent with BERT-style tokenization (expect 6–7 tokens including `[CLS]` / `[SEP]`).
4. For input that exceeds the model's max length (e.g., a 1000-word string), spike confirms truncation behaviour: either auto-truncates to 512 tokens, or returns an explicit error — documents which one happens.
5. For a batch of varied-length inputs, spike confirms padding behaviour: either auto-pads to the longest in batch, auto-pads to model max, or requires explicit `PaddingParams` configuration — documents which.
6. Spike compares token-ids for `"Hello World"` vs `"hello world"` — confirms whether the tokenizer is case-sensitive (bge-small-en-v1.5 should be uncased per the model card; verifying this against the actual config file we ship is the point of the spike).

**(c) On failure (any criterion not met):**
- Document the unexpected behaviour in the spike file's header comment.
- If the tokenizer.json from HuggingFace requires post-load configuration (explicit padding/truncation setup), update the Module breakdown with a `TokenizerConfig` struct + Phase 2 allocates time for it.
- If the tokenizer file we have is *not* the canonical bge-small one (e.g., we've grabbed an incompatible version), update Spike 3's "ONNX provenance" investigation to cover tokenizer provenance too, and document the canonical-source URL + SHA-256 in the spike file.
- Casing surprise (e.g., the tokenizer turns out to be cased when the model card says uncased) → blocker; tokenizer / model mismatch is exactly the pairing-failure class Q5's integrity check defends against, but if the canonical files are themselves mismatched we have a deeper provenance problem.

### Spike 3 — bge-small-en-v1.5 ONNX availability + license + provenance *(Medium risk — could force ADR-022)*

**Why:** Last known state — BAAI ships bge-small-en-v1.5 in safetensors format; ONNX exports exist in the community but provenance varies. If we ship a community-converted ONNX, we own the conversion provenance. If the source disappears, we need to be able to reproduce. This is a one-time investigation that locks in the model-supply-chain story before the integrity-check hashes (Q5) freeze the bytes we ship.

**(a) Artefacts:**
- A short markdown brief: `crates/vault-embedding/MODEL_PROVENANCE.md` (committed long-term — ~1 page). Documents: source URL of the ONNX file, who converted it (BAAI official vs community vs us), SHA-256 of the file as bundled, license confirmation, conversion-script reproduction notes if community-converted.
- IF community-converted: `scripts/convert-bge-small-to-onnx.py` (a reproducible PyTorch → ONNX export script using `optimum` or `torch.onnx.export`). Committed long-term so we can re-derive if the upstream disappears.
- IF official BAAI ONNX exists: just the brief + URL pointer. No conversion script needed.

**(b) Acceptance criteria (all must hold for spike PASS):**
1. The brief documents a definitive answer to: "Where does the ONNX file we ship come from?" (BAAI official / HuggingFace community user X / our own conversion).
2. License of the bundled ONNX is confirmed MIT (matching the safetensors original).
3. SHA-256 of the bundled ONNX is recorded — this becomes the value compiled into `BGE_SMALL_EN_V1_5_MODEL_SHA256` at Phase 2.
4. If community-converted: the conversion is reproducible from the script (verified by running it on a clean checkout and confirming the output ONNX matches the bundled SHA-256, OR documenting why bit-exact reproduction isn't possible — e.g., PyTorch nondeterminism — and proposing an alternative integrity story).
5. The chosen source URL is captured in MODEL_PROVENANCE.md so a future rebuild / audit can re-fetch.

**(c) On failure:**
- No clean source available + can't reproduce conversion → escalate. We may need to do our own conversion from BAAI's safetensors (more complex; raises the integrity-story bar). Land ADR-022 documenting the conversion chain in detail.
- License unclear or non-MIT on the ONNX file we'd ship → blocker; cannot redistribute. Need a different model file (BAAI's own ONNX export request, or a different MIT-licensed embedding model entirely).
- Multiple "official-looking" community conversions with different SHA-256s → pick one based on download count + author reputation, document the choice in MODEL_PROVENANCE.md, set the integrity hash to that one.

**Why:** ADR-012 established the precedent — when adding a heavy dep that pulls in a large transitive tree, audit the default features. - What each default feature pulls into the dep graph (`cargo tree -p vault-embedding -i <suspect-crate>`).
- Whether any default features are unwanted for V0.1 CPU-only inference.
- The chosen feature set (likely `default-features = false` + just `load-dynamic`).

**(b) Acceptance criteria (all must hold for spike PASS):**

**(c) On failure:**

**Why spike-before-lock matters here (recap with concrete precedent).** Two prior cases saved multi-day refactors / silent-failure bugs:
- **ADR-008 dryoc spike (~90 min):** discovered the BRD's sketch (`crypto_aead_xchacha20poly1305_ietf`) doesn't exist in dryoc 0.7 — `DryocStream` is the actual API. Implementation-without-spike would have hit the wall on day 1 of T0.2.9.

**What stays out of spike scope:** anything the spikes don't directly verify (e.g., perf vs the BRD §5.3 budget, batch throughput, multi-thread session safety) gets validated during the test phase, not pre-spike. Spikes are for de-risking *unknown-unknowns* in foreign APIs; perf benchmarks are for confirming *known-knowns* meet known targets — those belong in tests, not spikes.

## Notes for T0.1.11 (Tauri shell — bundling concerns inherited from T0.1.7)

T0.1.7 ships the embedding crate but NOT the application shell that wires it into a runnable binary. T0.1.11 (vault-tauri minimal UI + lib→bin conversion per ADR-003) inherits a non-trivial bundling story from this task. Flagged here so T0.1.11 starts from "implement this contract" rather than "discover this contract" — same pattern as ADR-015's consolidator-policy spec landing during T0.1.5.

**Bundle size budget impact (~200 MB delta vs current binary):**
- bge-small-en-v1.5 ONNX (FP32): ~133 MB
- Tokenizer files (`tokenizer.json` + any auxiliary BPE/wordpiece files): ~3 MB
- ORT dylib per platform: ~30–50 MB each (`libonnxruntime.dylib` macOS / `onnxruntime.dll` Windows / `libonnxruntime.so` Linux). Per-platform installer ships only the matching one.
- Total per-platform installer growth: ~170–185 MB. Current vault-tauri target binary is small (low single-digit MB once it converts to bin); this is a 50× growth event for the installer. Founder dogfood is fine (one machine, fast disk); customer-facing installers (V1.0) need a download-progress UX.

**Tauri config touchpoints (all in `crates/vault-tauri/tauri.conf.json` at T0.1.11):**
- `bundle.resources` array: lists the model file + tokenizer file + ORT dylib paths to include in the platform installer.
- `bundle.identifier` + signing config (BRD §11.7.5): the bundled dylib + model files are inside the signed artefact. macOS notarization scans every binary; Windows signtool covers the dylib. Adds ~30 sec to signing time per platform — worth flagging in CI runtime estimates at T0.2.x release-prep.
- macOS-specific: `Resources/` bundle directory layout, `tauri::PathResolver::resolve_resource` for runtime path lookup.
- Windows-specific: resources land alongside the `.exe` in a flat layout, different path resolution code from macOS — `tauri::PathResolver` abstracts this, but the dylib loading via ORT's `load-dynamic` still needs the absolute path resolved correctly per platform.
- Linux-specific (V0.2+ if AppImage): resources embedded in the AppImage; runtime extracts to `$APPDIR`.

**Code-signing scope expansion:**
- macOS: every binary inside the `.app` bundle gets signed individually (Apple notarization requirement). Adds the ORT dylib to the sign-list.
- Windows: signtool over the entire installer + the unpacked dylib at install-destination time.
- The model + tokenizer files are *not* executable so they aren't independently signed, but they ARE inside the signed installer so tampering at distribution time is detected at install time. Q5's runtime SHA-256 check is the second-line defence (catches post-install tampering or substitution).

**Runtime path resolution (the actual code T0.1.11 writes):**
- vault-app (T0.1.10) gets a config-overridable model directory + ORT dylib path, falling back to "alongside the binary" defaults.
- vault-tauri (T0.1.11) wires `tauri::PathResolver::resolve_resource` into the vault-app config layer at startup. Three paths to resolve: model.onnx, tokenizer.json, libonnxruntime.{dylib,dll,so} (with the right extension per platform).
- Dev-mode (`cargo tauri dev`) vs production-mode path differences: in dev, resources resolve relative to the source tree; in production, relative to the installed bundle. `tauri::PathResolver` handles the difference; T0.1.11 just needs to NOT hardcode any paths.

**T0.1.11 should also:**
- Add the ADR-010 alpha banner work that T0.1.4 deferred to T0.1.11 (already in HANDOFF.md as a pre-committed deliverable).
- Surface a one-time fatal-error dialog for the integrity-failure path locked in Q5 above.
- Verify the bundle on every target platform's installer, not just dev-mode. (Bundling bugs only surface in actual installers.)

**Why this lives in T0.1.7's plan, not as a separate doc:** the bundling concerns ARE T0.1.7 outputs from T0.1.11's perspective — the model file, tokenizer file, ORT dylib, and integrity-failure-dialog all originate in this task's design. Documenting them inline keeps T0.1.11's planning starting point clean: "implement these inherited contracts" beats "rediscover what T0.1.7 implies for me."

## Spike findings amendment block (v1.2 — applied 2026-04-30)

Spikes 1, 2, 3 ran via web research methodology; Spike 4 folded into Phase 1's first build. Methodology declared retroactively (going forward, methodology is declared per-spike at plan-draft time per the saved feedback rule "Spike methodology explicit per spike"). Findings:

**Spike 1 — `ort 2.0` API (web research, runtime confirmation = Phase 1 first build).** `Session::builder()?.commit_from_file(path)?` is the load API. `session.run(ort::inputs!["name" => tensor])?` returns `SessionOutputs`. `Tensor::from_array((shape, vec))` constructs inputs. `Tensor::extract_array()` (with `ndarray` feature) returns `ArrayViewD<'_, T>`; `extract_tensor()` returns `(&Shape, &[T])`. **`load-dynamic` mechanism: `ort::init_from(dylib_path)` MUST be called before any other ort use.** `ORT_DYLIB_PATH` env var works as a fallback. Idempotency requires `std::sync::Once` wrapping (Phase 1 implementation note above). PASS.

**Spike 2 — `tokenizers` crate (web research).** `Tokenizer::from_file(path)` loads tokenizer.json. `Encoding::get_ids() -> &[u32]`, `get_attention_mask() -> &[u32]`, `get_type_ids() -> &[u32]`. Manual `encoding.pad()` + `encoding.truncate()` for length control — `tokenizer_config.json`'s `model_max_length: 512` is NOT auto-applied by the Rust crate (Python tokenizers behaves differently). **Two implementation notes added to Phase 3: (a) `u32 → i64` conversion at the tokenizer/ort boundary, (b) manual `encoding.truncate(512, 0, TruncationDirection::Right)` after `encode`.** PASS.

**Spike 3 — bge-small-en-v1.5 ONNX provenance (web research).** Official BAAI ONNX export confirmed at `https://huggingface.co/BAAI/bge-small-en-v1.5/tree/main/onnx/model.onnx`. **Canonical SHA-256: `828e1496d7fabb79cfa4dcd84fa38625c0d3d21da474a00f08db0f559940cf35`** (133 MB, FP32, MIT licence, commit `5c38ec7` by Shitao). Tokenizer.json commit hash `9b09f79fe618f3969e13e52f12c01de64145ebc0` (SHA-256 captured at download time in Phase 1). **Critical secondary finding from `1_Pooling/config.json`: `pooling_mode_cls_token: true` — bge-small uses CLS-token pooling, NOT mean pooling.** This is a silent-failure-class bug if mis-implemented (output shape + L2 norm are identical between CLS and mean pooling; only semantic correctness differs, which the original test plan would not have caught). Test 9 (`embed_uses_cls_pooling_not_mean_pooling`) added to enforce the contract. ADR-022 NOT needed (no conversion provenance to document). PASS.

**Spike 4 — ort default features audit (deferred to Phase 1's first build).** Folded into Phase 1 per session decision. Conditional ADR-021 lands in Phase 1's commit if minimisation happened.

**Methodology note (going forward).** All three executed spikes were web research, decided unilaterally mid-execution. Going forward, every spike in a spike-before-lock list declares its methodology upfront alongside artefact + acceptance + failure mode (per saved feedback rule). For T0.1.7, web research was the right call: API-shape questions (Spikes 1+2) were authoritatively answered by docs.rs; configuration-metadata questions (Spike 3 — pooling mode) were ONLY discoverable via documentation, not runtime — a runtime spike with mean-pooling code would have produced output of shape (1, 384) with unit L2 norm and confirmed nothing about CLS-vs-mean correctness. Deferred runtime confirmation = Phase 1's first cargo build (real, not hand-waved).
