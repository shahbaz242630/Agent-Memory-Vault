## 1. Goal

Build `vault-retrieval` for V0.1: a single-strategy (Semantic) retriever that takes a natural-language query and returns the most relevant memories from the vault, filtered by mandatory boundary access control, with retrieval audit logging to the local audit chain. This is the user-facing "search" capability in BRD §6 V0.1 Definition of Success.

V0.1 scope is intentionally narrow: only the Semantic strategy. Graph, Temporal, Keyword, Frequency strategies, the QueryClassifier, and the Reranker land at T0.2.7. The discipline now is to shape the V0.1 trait + module structure so T0.2.7 is **purely additive**.

## 2. BRD §5.5 reconciliation

The BRD §5.5 sketch (lines 754–789) shows the full multi-strategy shape. For V0.1 we implement only what T0.1.10 / T0.1.9 / T0.1.11 actually consume.

Three reconciliation points where BRD literal collides with established invariants:

1. **Boundary mandatory, not Optional.** Sketch shows `Option<Boundary>`. ADR-005 / ADR-010 / ADR-015 establish a clear pattern: every access-control surface takes `&[Boundary]` mandatorily, empty → empty result (not error).
2. **`time_range` and `strategies` (the enum) deferred.** Both belong to the multi-strategy world. Including them with no implementation is "speculative scaffolding."
3. **`MultiStrategyRetriever` / `Reranker` / `QueryClassifier` not built.** Their absence in V0.1 is the no-scaffolding discipline.

**Forward-compat decisions in v1.2** (each tagged with the concrete-vs-hypothetical test from saved feedback memory):
- ✅ `RetrievalOptions { score_threshold, include_archived }` — concrete future use (T0.2.x consolidator WILL produce superseded memories; T0.2.7 reranking WILL want score thresholds)
- ✅ `MetadataStore::get_memories_batch` — concrete future use (T0.2.7 reranking WILL fetch top-50 candidates)
- ✅ `AuditAction::RetrievalQuery` variant — current need (audit chain logs every retrieval locally for the user's own forensic record, V0.1 onwards)
- ❌ ADR-021 retrieval-audit-privacy salt + hashing — **REMOVED in v1.2**, hypothetical future use only (cross-vault audit-log leak doesn't exist in V0.1 OR V0.2's planned distribution; defer to whenever audit-log distribution actually changes)

## 3. Pre-design questions (each carries a recommended answer)

### Q1 — Boundary contract: `&[Boundary]` mandatory, empty → empty result *(unchanged since v1.0)*

**Recommendation:** `RetrievalQuery.authorized_boundaries: Vec<Boundary>` (mandatory, owned). Empty Vec returns `Vec<RetrievedMemory>::new()` without round-trip to `EmbeddingProvider` or `VectorStore`. **Audit event still appends** with `result_count = 0` and `boundary_count = 0` — empty-auth is a legitimate audit data point.

### Q2 — Query-text validation policy *(unchanged)*

**Recommendation:**
- Trim leading/trailing whitespace before any other processing.
- Reject empty / whitespace-only queries with `VaultError::InvalidInput`.
- Cap query length at **2,048 chars**. Reject longer with `InvalidInput`.
- Reject queries containing ASCII control chars.

### Q3 — Result-limit defaults & caps *(unchanged)*

**Recommendation:** `RetrievalQuery.max_results: usize` mandatory. Retriever rejects `max_results == 0` and `max_results > 100`. Caller documentation: MCP default = 10, UI default = 20.

### Q4 — Relevance-threshold policy *(unchanged since v1.1)*

**Recommendation:** `RetrievalOptions.score_threshold: Option<f32>`. V0.1 default `None`. When `Some(t)`, drop results with `score < t`. The field exists for forward-compat; T0.2.7 reranking will tune defaults.

### Q5 — `time_range` and `strategies` field handling *(unchanged since v1.1)*

**Recommendation:** **Omit `time_range` and `RetrievalStrategy` enum + `strategies` field from V0.1.** Add additively at T0.2.7.

### Q5b — `include_archived` semantics *(unchanged since v1.1.1)*

**Recommendation:** Add `RetrievalOptions.include_archived: bool` with default `false`. Documented semantics: archived = superseded-by-consolidator OR user-archived (V0.2); superseded counts as archived for retrieval purposes; V0.1 is a no-op (no archived memories exist) but the SQL filter `AND superseded_by IS NULL` is wired from day one. Test fixture uses test-only `INSERT INTO memories ... superseded_by = '<uuid>'` to pin the V0.2 contract in V0.1.

### Q6 — `RetrievedMemory.explanation` string format *(unchanged)*

**Recommendation:** Fixed format: `"semantic: cosine={score:.4} (rank {rank}/{total})"`.

### Q7 — Score semantics: cosine similarity in [-1, 1], not LanceDB distance *(unchanged since v1.1)*

**Recommendation:** Score = cosine similarity in [-1, 1], higher = better. Transform `score = 1.0 - lance_distance`. See §6 for spike pre-resolution evidence.

### Q8 — Empty-query / no-vault-data / partial-result handling *(unchanged)*

**Recommendation:**
- Empty query (post-trim) → `VaultError::InvalidInput`. Audit event records the failure.
- Empty vault → `Ok(vec![])`. Audit event records `result_count = 0`.
- LanceDB returns < `max_results` → return what's there.

### Q9 — Result-order contract *(unchanged since v1.1.1)*

**Recommendation:** Score descending (best match first), equal-score tiebreak `memory.created_at DESC`. Locked in code-comment + trait-level property test.

### Q10 — Memory hydration: `MetadataStore::get_memories_batch` *(unchanged since v1.1)*

**Recommendation:** Lands in T0.1.8 directly (concrete future use: T0.2.7 reranking fetches top-50 candidates). Order-preserving; missing-id `warn!` and omit.

### Q-3.5 — Retrieval audit logging *(v1.2: SUBSTANTIALLY SIMPLIFIED — ADR-021 reversal)*

**v1.2 decision:** retrieval audit logging stays in V0.1; **the privacy-against-cross-vault-comparison machinery (per-vault salt + BLAKE3 query hash + migration 0003) is REMOVED.** Defer to whenever audit-log distribution actually changes — concrete-vs-hypothetical-future-use test fails for the salt scheme.

**What v1.2 keeps:**

1. **Audit event variant `AuditAction::RetrievalQuery`** in vault-core's audit-event enum. Joins the existing `memory.create / memory.update / memory.delete / cascade.* / store.corruption` family. Current need: every retrieval is recorded in the local audit chain for the user's own forensic record.

2. **Audit event details_json** (canonical sorted-key JSON per BRD §11.9.2 / T0.1.3):
   - `query_length: u32` — byte length of the post-trim query text
   - `boundary_count: u32` — `authorized_boundaries.len()`
   - `result_count: u32` — `RetrievedMemory` count returned
   - `max_results: u32` — caller-requested cap
   - `score_threshold: f32` (or `null` if `None`) — caller's threshold
   - `include_archived: bool` — caller's archive option
   - `latency_ms: u64` — wall-clock from `retrieve()` entry to audit-append (excluding the audit-append itself)
   - `error: String` (only present on `result = error` rows; e.g., `"InvalidInput: query empty after trim"`)

   **NOT included:** `query_hash`, salt-derived anything, query plaintext.

3. **`SemanticRetriever` writes the audit event** via the `Arc<MetadataStore>` it already holds for memory hydration (Q10). The audit-append happens **after** the retrieval completes (success OR error path), as a single audit-only append (no need to atomic-bundle with retrieval — retrieval is read-only against vector + metadata stores, the audit chain's BLAKE3 chain integrity is maintained per T0.1.3).

**What v1.2 explicitly DROPS (deferred to whenever audit-log distribution actually changes — likely never per current V0.2 plan):**

| Dropped | Why |
|---|---|
| ADR-021 entirely | Hypothetical threat (cross-vault audit-log leak) doesn't exist in V0.1 or V0.2 |
| Migration `0003_vault_metadata.sql` | No salt to persist |
| `vault_metadata` SQLite table | No salt to store |
| `MetadataStore::retrieval_audit_salt()` accessor | No salt to read |
| Salt load-or-init logic in `MetadataStore::open` | Removed |
| `info!("vault salt initialised")` first-open log | Salt no longer exists |
| `BLAKE3(salt ‖ query)` query hashing | No salt → no hash |
| `query_hash: [u8; 32]` in audit details_json | Removed |
| 4 salt-related tests (salt persistence, deterministic-per-salt, divergent-across-vaults, cross-vault hash divergence) | Removed |

**What changes in V0.2+ if/when audit-log distribution ever lands:** at that point the design conversation includes salt + hashing + migration as part of the same threat-model decision, with knowledge of what the actual sync mechanism looks like. The cost of adding the salt later when audit logs ever distribute is the same migration cost as adding it now, plus we'll know the threat we're defending against.

**Threat-model honesty:** the V0.1 audit chain is local-only and SQLCipher-encrypted. An attacker with read access to the audit chain has already compromised the master key (unlocked the SQLCipher DB), at which point everything else is also compromised. Defense-in-depth against a hypothetical cross-vault-leak future doesn't add real protection in V0.1's threat model.

## 6. Spike check

**Q7 spike — PRE-RESOLVED via codebase re-read 2026-05-01 (carried forward from v1.1.1 §6, unchanged in v1.2).**

Confirmed in code (no web research run):
- `vault-storage/src/vector_store.rs` lines 103–105: *"Returned scores are distances under the configured metric (cosine for V0.1; smaller = closer)."*
- `vault-storage/src/vector_store.rs` lines 184–186: *"Returned scores are distances (smaller = closer; identical vectors → 0)."*
- Implementation site lines 357 + 375–386: explicit `.distance_type(DistanceType::Cosine)` + extracts `_distance` column.

**Confirmed:** `LanceVectorStore::search` returns `(MemoryId, distance)` where distance is cosine distance from `DistanceType::Cosine`, in `[0, 2]`, smaller = closer. The Q7 transform `score = 1.0 - distance` produces cosine similarity in `[-1, 1]`, higher = better.

**Action item in `SemanticRetriever::retrieve`** (carried from v1.1.1, unchanged):
```rust
// LanceDB's DistanceType::Cosine returns cosine *distance* (smaller = closer;
// identical vectors → 0). See vault_storage::vector_store::VectorStore::search
// doc-comment for the source-of-truth contract. We transform to cosine
// similarity in [-1, 1] (higher = better, conventional IR mental model
// per T0.1.8 plan §3 Q7 + Q9).
let score = 1.0 - distance;
```

**Per the saved-feedback rule** (web-research spikes need runtime confirmation in the next phase): Phase 2's first cargo test run against real `LanceVectorStore::search` IS the runtime confirmation — test 1 (happy-path top-2 cats), test 7 (result-order contract), test 8 (score range) all fail loudly if the transform is wrong.

**Result:** no plan changes from spike. v1.2 is final pending no surprises in re-review.
