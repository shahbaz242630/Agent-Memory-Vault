## Three pre-design confirmations (locked before any C1 code)

### Confirmation 1 — Connection pattern: extend `MetadataStore` with `with_transaction<F, T>`

**Decision.** Phase C1 adds **`MetadataStore::with_transaction<F, T>(&self, f: F) -> VaultResult<T>`** at `pub(crate)` visibility. Closure receives `&rusqlite::Transaction` (not `&mut Connection`) so callers cannot accidentally `COMMIT` / `ROLLBACK` from inside — the helper does both. Closure is sync (runs inside `spawn_blocking`).

```rust
impl MetadataStore {
    pub(crate) async fn with_transaction<F, T>(&self, f: F) -> VaultResult<T>
    where
        F: FnOnce(&rusqlite::Transaction<'_>) -> VaultResult<T> + Send + 'static,
        T: Send + 'static,
    {
        let inner = self.inner.clone();
        tokio::task::spawn_blocking(move || {
            let mut conn = inner.lock()?;
            let tx = conn
                .transaction()
                .map_err(|e| VaultError::Storage(format!("begin tx: {e}")))?;
            let out = f(&tx)?;
            tx.commit()
                .map_err(|e| VaultError::Storage(format!("commit: {e}")))?;
            Ok(out)
        })
        .await
        .map_err(|e| VaultError::Storage(format!("spawn_blocking join: {e}")))?
    }
}
```

**Reasoning.** The orchestrator's user-write path is one atomic SQLite transaction across **three tables**:
1. `INSERT INTO memories ...` (source of truth).
2. `INSERT INTO audit_log ...` (BLAKE3-chained event, capturing `seq` for the next step).
3. Either `INSERT INTO retry_queue (memory_id, sequence_id = audit_seq, ...)` if queue under cap, or `INSERT … ON CONFLICT INTO pending_sync` if at cap.

All three commit together or none do. Without `with_transaction`, the orchestrator would either (a) open its own SQLite Connection (multi-Connection SQLCipher in WAL mode is a known footgun — keys re-derive per Connection, locking gets weird) or (b) hold three separate sub-transactions and try to coordinate them (loses atomicity). `with_transaction` borrows MetadataStore's existing `Mutex<Connection>` for the whole sequence — single Connection, single Mutex, atomic guarantee.

**Phase B's `with_conn_blocking` is a strict subset** — it locks the connection but doesn't begin a transaction. Phase B's three modules (each CRUD-on-its-own-table) use `with_conn_blocking`. Phase C's cross-table writes use `with_transaction`. Both helpers stay `pub(crate)`; orchestrator and the three Phase B modules are the only callers.

**Trade-off accepted.** SQL inside `with_transaction` runs on a single thread (the spawn_blocking worker holding the mutex). For V0.1's expected throughput (handfuls of writes/sec) this is fine — same trade-off as the rest of `MetadataStore`. A connection pool is a V1.0 concern.

### Confirmation 2 — Background task lifecycle: `step()` + `run(rx)` separation

**Decision.** Each background worker exposes **two entry points**, with the cancel signal carried by `tokio::sync::watch::Receiver<bool>` (no new workspace dep — `watch` is already in `tokio` we depend on):

```rust
pub struct RetryWorker {
    queue: RetryQueue,
    dead_letter: DeadLetter,
    backend: Arc<StorageBackend>,
    jitter: Box<dyn JitterSource>,
    poll_interval: Duration,
}

impl RetryWorker {
    /// Run exactly one iteration: poll due, attempt one entry, persist
    /// outcome. Synchronous (in the async sense — returns when one unit
    /// of work is done). Used by tests to drive the worker deterministically
    /// without spawning a tokio task.
    pub async fn step(&mut self) -> VaultResult<StepResult>;

    /// Production loop: until cancel signal flips to `true`, alternate
    /// `step().await` + `sleep(poll_interval)` (with sleep interruptible
    /// by the cancel signal so shutdown doesn't wait out the full delay).
    /// Honors graceful shutdown — finishes the current step before exiting.
    pub async fn run(mut self, mut cancel: tokio::sync::watch::Receiver<bool>);
}

#[derive(Debug, PartialEq)]
pub enum StepResult {
    /// No entries due; worker should sleep until poll_interval.
    Idle,
    /// Worked one entry to success.
    SucceededEntry { id: Uuid, attempts_made: u32 },
    /// Worked one entry to a reschedule (failure, retries remaining).
    Rescheduled { id: Uuid, next_attempt_at: DateTime<Utc> },
    /// Worked one entry to dead-letter (exhausted or permanent).
    DeadLettered { id: Uuid, reason: DeadLetterReason },
}
```

The `run` loop pattern:

```rust
loop {
    if *cancel.borrow() { break; }
    let _ = self.step().await; // log errors but keep going
    tokio::select! {
        _ = tokio::time::sleep(self.poll_interval) => {},
        _ = cancel.changed() => { break; }
    }
}
```

**Phase C1 does NOT spawn the worker.** `T0.1.10 (vault-app::Application::start)` will create the `watch::channel(false)`, call `tokio::spawn(worker.run(rx))`, and store the JoinHandle + the sender for shutdown.

**Reasoning.** Two clean benefits:
1. **Tests drive `step()` directly** — no `tokio::spawn`, no fake clock, no flaky timing assertions. The 5 adversarial tests in Q5 of Phase A all become "set up state, call `step()` N times, assert outcomes." Deterministic.
2. **Application owns lifecycle** at T0.1.10 — Phase C doesn't preempt that. The worker struct is a value, not a running task. Easy to mock in vault-app's wiring tests too.

**Why `watch::channel(false)` and not `tokio_util::CancellationToken`?** `tokio` is already pinned in workspace deps; `tokio::sync::watch` is part of it. `tokio-util` would be a new workspace dep just for one type. The `watch` ergonomics are slightly less polished (`*rx.borrow()` vs `token.is_cancelled()`) but the savings on supply-chain surface + audit churn beat the ergonomic loss for V0.1's needs. Adopt Shahbaz's smaller-item-(c) recommendation.

**Same shape for the divergence detector in C2** — `step()` runs one detection pass; `run(rx)` is the production loop (V0.1: only invoked on startup + on-demand via CLI; the time-based daemon mode is a V0.2 concern per Phase A Q3).

## One-row-per-write retry model (Issue 1 resolution)

**Decision.** Adopt **one row per cascading write**, with the row's `payload` carrying everything both stores need (`embedding` for LanceDB, `entities` / metadata for DuckDB). The retry worker's `step()` runs both store ops idempotently in sequence; either failure → whole entry reschedules. Dead-letter granularity = whole memory.

**Phase B's `CascadeOperation` enum changes:**

```rust
// Before (Phase B as shipped at 2c04fdb):
pub enum CascadeOperation {
    LancedbWrite, LancedbUpdate, LancedbDelete,
    DuckdbWrite, DuckdbUpdate, DuckdbDelete,
}

// After (C1):
pub enum CascadeOperation {
    Write,
    Update,
    Delete,
}
```

Persisted strings: `"write" / "update" / "delete"`. Phase B's six per-store strings are no longer valid — but **no production data exists yet** (V0.1 is local-only, pre-dogfood; every Phase B test creates a fresh tempdir). So this is a code-only change, not a data migration. Phase B's test sites that reference `CascadeOperation::LancedbWrite` etc. update mechanically (~10 references across `retry_queue.rs`, `dead_letter.rs`, `pending_sync.rs` test modules).

**Walking through Shahbaz's three pushback questions:**

1. **What does retrieval see during the divergence window?** V0.1 retrieval is **LanceDB-only** (T0.1.8 ships "Semantic Strategy Only"). DuckDB isn't on the retrieval path until V0.2+ graph traversal. So with two-rows, "lancedb succeeded but duckdb still pending" was invisible to V0.1 retrieval — the partial-availability was an empty benefit.
2. **What does "lancedb dead-lettered but duckdb succeeded" mean to a user?** A confusing message ("did the memory write or didn't it?") for a state retrieval can't actually use. Worse mental model than "the memory failed to fully cascade."
3. **Why is independent dead-lettering valuable?** It isn't, given (1) and (2). My v1 motivation was implementation flexibility ("worker can drain ops independently"), but that's overhead in service of nothing.

**Precondition this exposes (load-bearing).** With one-row + lockstep-or-fail, the worker's permanent-failure paths (`DimensionMismatch / AccessDenied`) become *bug surfaces* rather than expected behaviours. The orchestrator's `write_memory` MUST validate at entry, before SQLite commit:

- **Embedding dim** matches `LanceVectorStore::dimension()`. Mismatch → reject with `VaultError::DimensionMismatch` *before* the SQLite write.
- **Boundary** is in the caller's authorized set. Mismatch → `VaultError::AccessDenied`.

If a write reaches the queue, the only viable failure is transient. A permanent classification surfacing in the worker means a bug — dead-letter loud + audit event tagged `unexpected_permanent_in_worker`. (Schema-mismatch `Storage(msg).contains("schema")` could still legitimately surface late if the schema drifts mid-session, but that's rare and equally affects both stores.)

**Asymmetric refactor cost (Shahbaz's framing).** One-row → two-rows later is mechanical (split the enum, double-insert in `write_memory`). Two-rows → one-row later is a live data migration (collapse retry_queue rows by `memory_id`, handle in-flight retries). The asymmetry favours one-row as the conservative default unless retrieval needs partial-store availability — which V0.1 explicitly does not.

## Hard-corruption surface (Issue 2 resolution)

**Spike findings (verified 2026-04-30 via `crates/vault-storage/examples/lance_corruption_spike.rs`):** LanceDB 0.8 detects fragment corruption **lazily**:

- `LanceVectorStore::open()` → **Ok** (opens table metadata; no fragment scan).
- `count(None)` → **Ok(5)** — LanceDB stores row counts in a manifest, separate from fragment data. Metadata-only path, unaffected by data corruption.
- `search(...)` → **Err("invalid uuid in row 4: invalid length: expected length 32 for simple format, found 0")** — the row decoder hits the corrupted UUID column bytes.

Concrete corruption: overwrite first 64 bytes of `memories.lance/data/<uuid>.lance` with `0xAB`. Open + count both pass; search fails on first read.

**Adopting Option B from review:** C1 adds **explicit eager validation in `StorageBackend::open`**. `LanceVectorStore::open` stays minimal (just connects + opens table metadata); the validation lives one layer up at the orchestrator boot.

**Validation shape (C1 implementation contract):**

```rust
impl StorageBackend {
    pub async fn open(...) -> VaultResult<Self> {
        let metadata = MetadataStore::open(...).await?;
        let vector = LanceVectorStore::open(...).await?;   // metadata-only — succeeds even on hard corruption
        let graph = DuckDbGraphStore::open(...).await?;    // similarly minimal

        // Eager validation: probe each store with a small read so corruption
        // surfaces NOW, not the first time the user searches.
        match vector.validate_readable().await {
            Ok(()) => {},
            Err(e) => {
                // CRITICAL audit + degraded-mode flag (still return Ok so
                // the founder can run vault-cli dead-letter list / triage).
                metadata.append_audit_event(/* tag: lancedb_corruption_at_open, error: e */).await?;
                return Ok(StorageBackend { degraded: DegradedMode::LanceUnreadable, ... });
            }
        }
        // Same shape for graph.validate_readable().
        ...
    }
}
```

`VectorStore::validate_readable()` is added to the trait in C1: a default impl that runs a minimal scan (e.g., `count(None)` followed by a 1-row `query().limit(1).execute().try_collect()` — the count is metadata-cheap; the limit-1 read forces a fragment touch). For an empty store, the limit-1 read is a no-op success. For a corrupted-data store, the read errors as the spike showed.

**Why not fail `open()` outright on corruption?** Shahbaz's Phase A Change 1 explicitly wants degraded mode: "the system enters degraded mode (SQLite still accessible, LanceDB unavailable, vault-cli `dead-letter list` works for triage)." Returning `Err` from `open` would lock the founder out of the recovery tool. Returning `Ok` with `degraded: DegradedMode::*` set lets vault-cli function while signalling clearly via audit event + UI banner (UI banner lands at T0.1.11).

**Hard-corruption test (lives in C1, with a clear expected failure mode):**
- Setup: insert 5 memories, drop the store, corrupt a Lance fragment file's first 64 bytes.
- Open the `StorageBackend` again.
- Assert: `open()` returns `Ok`, `degraded == DegradedMode::LanceUnreadable`, audit log has a `lancedb_corruption_at_open` event with the underlying error attached, and `metadata.list_memories(...)` still works (the SQLite side is unaffected).

**Secondary finding worth noting in the plan.** Because `count(None)` is metadata-only, **the divergence detector's tier-1 count comparison cannot catch hard corruption** either — count would show 5 in both SQLite and LanceDB, but LanceDB's fragment data is unreadable. Tier-2 sampled-existence has the same blind spot (existence checks pass on corrupted-but-listed rows). So C2's divergence is the right surface for soft corruption (silent row drops, FS bit flips on metadata files); C1's eager-validation-at-open is the right surface for hard corruption (fragment data garbled). Two distinct failure classes, two distinct surfaces.

**Spike artifact retained.** `crates/vault-storage/examples/lance_corruption_spike.rs` stays in-tree as executable documentation of LanceDB's lazy-corruption behaviour. Re-runnable via `cargo run -p vault-storage --example lance_corruption_spike` to verify after future LanceDB bumps. Same long-term-doc pattern as `crates/vault-sync/examples/dryoc_spike.rs` (ADR-008). Lands in C1's commit alongside the orchestrator code that depends on its findings.

## Cascading orchestrator design (C1, key contracts)

### Write-path atomicity (the load-bearing flow)

```rust
impl StorageBackend {
    /// User-write entry point. Returns Ack as soon as SQLite + audit log
    /// commit. Cascading LanceDB / DuckDB writes happen async via the
    /// retry worker.
    pub async fn write_memory(&self, memory: &Memory, embedding: &[f32]) -> VaultResult<Ack>;
}

pub struct Ack {
    pub memory_id: MemoryId,
    pub sqlite_committed_at: DateTime<Utc>,
}
```

Inside `write_memory`, the orchestrator:

**1. Validates inputs eagerly (before any SQLite write):**
- `embedding.len() == self.vector.dimension()` else `VaultError::DimensionMismatch`.
- `memory.boundary` is in the caller's authorized set else `VaultError::AccessDenied`.

(Phase A Q1's permanent-failure classes become impossible-to-reach-the-queue by construction.)

**2. Runs ONE `with_transaction` call:**

a. `INSERT INTO memories ...` (atomic with b–c).
b. `INSERT INTO audit_log ...` and capture the resulting `seq` (audit chain's monotonic index — the source of the cascade-ordering invariant per ADR-017).
c. Decide cap: if `retry_queue` row count `< 10_000`, `INSERT INTO retry_queue` with **a single row** carrying `sequence_id = audit_seq`, `operation = Write` (or `Update` / `Delete`), and a payload containing both `{embedding, boundary, entities, ...}`. Else: `INSERT … ON CONFLICT(memory_id) DO UPDATE` into `pending_sync` once per memory.
d. `COMMIT`.

**Rationale: always-enqueue, never-synchronous-cascade.** The worker is the sole path that touches LanceDB / DuckDB. User writes never block on those stores. The retry queue's FIFO-per-memory discipline (sequence_id ASC) is naturally correct because audit `seq` is monotonic per SQLite txn-commit order. This matches Phase A Q1 verbatim and is locked in ADR-017.

**Updates and deletes** follow the same pattern: SQLite-side mutate + audit append + enqueue one cascade row with `operation = Update` / `Delete`.

### Worker + dead-letter integration

`RetryWorker::step()` polls due, picks the lowest-`sequence_id` due entry, then **runs both store ops idempotently in sequence**:

1. Vector op (LanceDB upsert / update / delete based on `entry.operation`). Idempotent because `merge_insert(["id"])` is no-op on a re-run with identical input (T0.1.4 invariant).
2. Graph op (DuckDB create_entity / update / delete). Idempotent because Phase B's `create_entity` already uses `INSERT … ON CONFLICT DO NOTHING`-equivalent (T0.1.5 pre-flight pattern).

If both succeed → `RetryQueue::record_success(id)` deletes the row. If either fails → `RetryQueue::record_failure(id, err.to_string(), is_permanent(&err), &mut jitter)`. The returned `FailureOutcome::DeadLetter { entry, last_error, reason }` triggers `DeadLetter::insert(NewDeadLetter::from(entry, last_error, reason))` + a CRITICAL audit event. Reschedule path is silent in audit (we don't want every backoff attempt to spam the chain).

**Idempotency invariant for the worker is load-bearing.** If a previous `step()` succeeded on LanceDB but failed on DuckDB, the next `step()` re-runs the LanceDB op (no-op) before the DuckDB op (re-attempted). No partial-state tracking in the payload — the underlying upserts are the source of truth.

## ADR amendments landing in C1 (one commit)

Per smaller-item-(b), each ADR has a single subject for clean reference.

1. **ADR-009 amendment** — formalises everything Phase A Q2 already drafted: 8-attempt schedule (1/2/4/8/16/30/60/120s with ±25% jitter), `is_permanent` classifier set (`DimensionMismatch / AccessDenied / Storage(msg).contains("schema")`), pending_sync overflow path replaces the original "pause LanceDB/DuckDB writes" sketch. Lands as an amendment block under the existing ADR-009 in HANDOFF.md.

2. **New ADR-016 — Connection-ownership for the cascading orchestrator.** Orchestrator routes cross-table writes (memories + audit_log + retry_queue / pending_sync) through `MetadataStore::with_transaction<F, T>(&self, f: F) -> VaultResult<T>` rather than opening a second SQLite Connection. Closure receives `&rusqlite::Transaction<'_>` (not `&mut Connection`) so callers cannot accidentally `COMMIT` / `ROLLBACK`. Reasoning: SQLCipher in WAL mode is sensitive to multiple Connections re-deriving the key; single-Connection-via-Mutex is the established pattern (T0.1.3) and `with_transaction` extends it without parallel state. Per Confirmation 1 of this plan.

3. **New ADR-017 — Cascade-ordering invariant.** `retry_queue` is strict FIFO per `memory_id` by `sequence_id` ASC; concurrent writes to the same memory serialise by SQLite commit order via the audit chain's monotonic `seq`. The chain's `seq` is unique per atomic write (assigned inside the `with_transaction` `INSERT INTO audit_log`) so the `(memory_id, sequence_id)` UNIQUE on retry_queue (Phase A migration 0002) is the structural enforcement. Per Phase A Q1.

4. **New ADR-018 — `FullySynced` deferral + eager corruption validation.** Two related items deferred / locked here:
   - `FullySynced` future deferred to T0.2.x; `RetryQueue::is_fully_synced(memory_id) -> bool` is the V0.1 interim API (per Phase A Q1).
   - **Eager corruption validation in `StorageBackend::open`** (this doc, "Hard-corruption surface"). LanceDB detects fragment corruption lazily (verified by spike); the orchestrator's `open` calls `VectorStore::validate_readable` and `GraphStore::validate_readable` to surface corruption immediately as a CRITICAL audit event + degraded-mode flag. Documented as a deliberate decision: we own the eagerness, we don't rely on the underlying store to provide it.

5. **Tech-debt entry** — split `VaultError::Storage(String)` grab-bag into structured variants (`SchemaMismatch(...)`, etc.) so `is_permanent` becomes an exhaustive match. Logged but explicitly NOT done in T0.1.6 (no drive-by refactoring per CLAUDE.md).
