# T0.1.6 Implementation Plan — Cascading Storage Backend + Retry Queue

**Status:** Approved 2026-04-30 with three required changes + four clarifications applied. Implementation in flight: failing tests first, then code, then DoD gates.
**Scope:** vault-storage `StorageBackend` orchestrating cascading writes across `MetadataStore` (SQLite/SQLCipher), `LanceVectorStore` (LanceDB), `DuckDbGraphStore` (DuckDB), with persistent retry queue, dead-letter table, divergence detection. Per BRD §5.2, §6 V0.1, ADR-009.
**Author:** Claude (Opus 4.7), 2026-04-30

## Q1 — "Cascading write succeeded" semantics

**Decision.** SQLite synchronously acked + LanceDB and DuckDB queued asynchronously. Public API returns a single `Ack { memory_id, sqlite_committed_at }` value as soon as SQLite + audit log commit. Async-side cascading runs in a background task; failures land in the retry queue (Q2). **No `FullySynced` future in V0.1.**

**Cascade-ordering invariant (added per Shahbaz review).** The retry queue is **strict FIFO per `memory_id`**. Concurrent writes to the same memory serialise by **SQLite commit order**, even when the downstream LanceDB / DuckDB writes complete in a different filesystem-write order. This is the canonical ordering the rest of the system reasons against — a third reader observing LanceDB between two cascading writes sees them in commit order, not in completion order. Implementation: per-`memory_id` enqueue with a sequence number derived from SQLite's audit-event chain index (which is monotonic per the BLAKE3 audit chain from T0.1.3). Documented in `cascading.rs` as a load-bearing invariant; tested in Q5 test 4.

**Reasoning.** SQLite is the source of truth (ADR-007 + audit chain). Once SQLite + audit are committed, the memory is durably recorded with full provenance — losing LanceDB / DuckDB rows is a *retrieval-quality* failure (eventually consistent), not a *data-loss* failure. V0.1's only callers (UI add, MCP write, search) don't care about full-sync state; the retry queue handles eventual consistency.

**Alternatives considered.**
- *Synchronous all-three-or-fail.* Rejected: any LanceDB / DuckDB hiccup (FS flush, brief lock) would block every user write even though SQLite already has the data.
- *Add `FullySynced` future now (Shahbaz's opening position).* Pushed back. The only V0.1 caller that *might* need it is the consolidator (BRD §5.6), which lands in T0.2.x. Building the synchronisation primitive (oneshot channel keyed by memory_id, plus dead-letter → `Err` translation, plus max-wait timeout) and not exercising it is exactly the speculative-scaffolding the CLAUDE.md hard rules forbid. **Defer to T0.2.2 (consolidator phase 1)** where we'll know the exact contract the consolidator needs. The retry queue is queryable in V0.1 (`is_fully_synced(memory_id)` returns `bool`) — that's enough to bolt the future on later without reshaping the cascade.

**Trade-offs accepted.** Cross-device sync (V0.2) cannot trust "the cascade finished" until queue is drained. T0.2.x sync engine will need to add the `FullySynced` primitive itself — slight rework cost. Worth it vs building it speculatively now.

## Q2 — Retry queue policy

**Decision.** Bounded queue at 10,000 entries + exponential backoff with jitter (base 1s, ×2, cap 5min) + **8 max attempts** before dead-letter + 1 worker (single retry loop). **Amends ADR-009** (which said 5 attempts, ~4 min span) — explicit amendment landed in HANDOFF.md as part of T0.1.6.

**Reasoning.** Walking through the failure shapes:
- *Transient FS hiccup* (antivirus scan, brief lock contention): typically resolves within 10–30s. 8 attempts spanning 1+2+4+8+16+30+60+120 ≈ 4 min handles these.
- *Structural failure* (corruption, schema drift, FS quota): no number of retries helps. 8 vs 5 vs 15 attempts costs ~4 min more wall time before the user-visible "needs attention" signal — acceptable.
- *Cloud-sync flake* (V0.2 onward): N/A in V0.1, and V0.2 will revisit this number anyway.

**Alternatives considered.**
- *15 attempts spanning ~80 min (Shahbaz's opening position).* Pushed back. 80 min is too long to surface a structural failure; if LanceDB has corrupted itself, we want the founder to know within minutes, not over an hour. Network-drive-like transient classes (where 15-attempt forgiveness shines) aren't part of V0.1's local-only profile.
- *5 attempts (ADR-009 original).* Rejected: ~4 min total span is right at the edge of typical FS hiccup duration. 8 attempts gives a safety margin without crossing into "user notices the lag."
- *Unbounded queue.* Rejected per ADR-009 reasoning (runaway disk consumption hides the real failure).

**Backoff schedule (8 attempts, all with ±25% jitter):** 1s, 2s, 4s, 8s, 16s, 30s, 60s, 120s. Total span ≈ 4 min worst case.

**Permanent-failure classification (added per Shahbaz review).** Not every error is worth retrying. The retry worker calls `is_permanent(&err)` on every failure; if `true`, the entry **dead-letters on attempt 1** instead of cycling through 8 backoffs. Permanent-failure set:

```rust
fn is_permanent(err: &VaultError) -> bool {
    match err {
        // Vector dimension mismatch — retrying with the same vector will
        // fail forever (LanceDB schema mismatch).
        VaultError::Storage(msg) if msg.contains("dimension") => true,
        // Boundary / access-control violations — retrying won't fix authorization.
        VaultError::AccessDenied(_) => true,
        // Schema-shape mismatch (e.g., constraint violations from a future
        // schema migration we haven't applied locally yet).
        VaultError::Storage(msg) if msg.contains("schema") => true,
        // Everything else: filesystem hiccup, lock contention, transient I/O —
        // retry per backoff schedule.
        _ => false,
    }
}
```

Rationale: `DimensionMismatch` and boundary violations are configuration / contract errors, not transient — wasting 4 min before surfacing them via dead-letter is pure latency. The classifier is intentionally conservative: anything we're not sure about retries (false negatives are recoverable; false positives lose retry-attempts but go to a queryable dead-letter row that the founder sees via vault-cli within seconds). String-matching on `VaultError::Storage(msg)` is a workaround for the current grab-bag variant; **future ADR**: split `VaultError::Storage(String)` into structured variants (`VaultError::DimensionMismatch { expected, got }`, etc.) so this classifier can use exhaustive match. Logged in HANDOFF tech debt.

**Cap behaviour at 10k entries (refined per Shahbaz clarification).** Three-tier behaviour:
1. SQLite write **always succeeds** (audit event recorded, memory committed).
2. The new entry is **NOT enqueued** to the retry queue (which is at cap). Instead, a row is inserted into a separate **`pending_sync` table** (schema below) marking the memory as needing async cascade once queue capacity is available.
3. CRITICAL log + audit event fired the first time the queue hits cap (and again on each "wave" — debounced, not on every overflow write — so logs don't get drowned).

```sql
CREATE TABLE pending_sync (
    memory_id  BLOB PRIMARY KEY REFERENCES memories(id),
    operation  TEXT NOT NULL,   -- 'lancedb_write' | 'duckdb_write' | 'lancedb_update' | 'duckdb_update' | 'lancedb_delete' | 'duckdb_delete'
    queued_at  TEXT NOT NULL    -- RFC3339, when the SQLite write completed
);
```

The divergence detector (Q3) treats `pending_sync` as a second input alongside its sampled-existence check: on every divergence run, it pulls from `pending_sync` (oldest first) and re-enqueues into `retry_queue` while the queue has capacity. Queue drained + `pending_sync` empty = system back to normal. This means cap overflow is **never** lost — it's transferred to a slower-but-deterministic catch-up path.

**Trade-offs accepted.** Slightly longer span than ADR-009's original (4 min vs ~4 min — same order, different attempt count). HANDOFF must explicitly amend ADR-009 to record the change.

## Q3 — Divergence detection schedule

**Decision.** **On startup + on-demand only for V0.1.** Drop the 24-hour timer until V0.2.

**Reasoning.** V0.1 is founder-only, manual entry, no sync. The founder closes/reopens the app frequently during dogfood (typical for power users on a Mac/Windows desktop). On-startup catches every drift introduced since the last close. A 24-hour wall-clock timer adds: a tokio interval task, careful semantics for "what if app was closed when timer fired," and an extra background thread, all to cover a window that startup-checks already cover for typical V0.1 usage. **Defer the timer to T0.2.x** when the app becomes more daemon-like (sync engine running) and the closed-app gap actually matters.

**Detection method.** Two-tier:
1. **Count comparison (cheap).** `SELECT COUNT(*) FROM memories WHERE valid_until IS NULL` in SQLite vs LanceDB row count vs DuckDB entity count. Counts not equal → tier 2. Counts equal → tier 2 still runs but on a smaller sample.
2. **Sampled-existence check.** Take 100 memory IDs sampled from SQLite via **deterministic stratified sampling** (refined per Shahbaz clarification): 50 from the most recent 30 days and 50 from older (or all if either window has fewer). The RNG is seeded from `current_date.timestamp() / 86400 * 0xDEADBEEF` (a daily-rotating seed): rotates daily so different rows get checked over time, deterministic-within-the-day so that two divergence runs on the same calendar day check the *same* sampled rows (lets us correlate findings across multiple checks within a day). Tests use a fixed override seed (`fn sample_with_seed(seed: u64)`) and assert on the exact rows sampled. For each sampled ID, verify presence in LanceDB and any associated entities in DuckDB. Missing = divergence finding.

Existence check, not content hash — content lives in SQLite only; embedding lives in LanceDB only; entities live in DuckDB only. Cross-store *content* comparison would require storing redundant data. Existence is the right invariant.

**Pending-sync sweep (added per Q2 cap-overflow refinement).** On every divergence run, the detector first sweeps the `pending_sync` table (oldest first) and re-enqueues into `retry_queue` up to the queue capacity. This is the catch-up path for memories that overflowed cap during a sustained store outage.

**Alternatives considered.**
- *24h timer + startup + on-demand (Shahbaz's opening).* Pushed back per above. Re-add in T0.2.x.
- *Every-Nth-write check.* Rejected per Shahbaz: amortised cost on the user-write path is the wrong place to put divergence work.
- *Weekly cadence.* Rejected per Shahbaz: too long for a memory product.

**Trade-offs accepted.** Long-running V0.1 sessions (founder leaves the app open for days) won't catch silent divergence between session-starts. Mitigated by: founder typically restarts daily, on-demand command available via `vault-cli divergence-check` (Q4).

## Q4 — Dead-letter user surface for V0.1

**Decision.** `dead_letter` table in SQLite (added by `0002_retry_dead_letter.sql` migration) + CRITICAL log on dead-letter event + audit event recorded + **`vault-cli` admin binary** with three subcommands. No UI in V0.1.

**Schema (final, refined from Shahbaz's opening position):**
```sql
CREATE TABLE dead_letter (
    id                       BLOB PRIMARY KEY,        -- DLQ entry ID, UUID v7
    memory_id                BLOB NOT NULL,           -- references memories(id)
    failed_operation         TEXT NOT NULL,           -- 'lancedb_write' | 'lancedb_update' | 'lancedb_delete'
                                                       -- | 'duckdb_write' | 'duckdb_update' | 'duckdb_delete'
    failure_reason           TEXT NOT NULL,           -- last error message (truncated to 4KB)
    attempts_made            INTEGER NOT NULL,
    first_failed_at          TEXT NOT NULL,
    last_attempted_at        TEXT NOT NULL,
    payload_format_version   INTEGER NOT NULL,        -- so we can evolve the payload shape
    payload                  BLOB NOT NULL,           -- JSON-serialised retry context
    resolution               TEXT NULL,               -- NULL = pending
                                                       -- 'retried_succeeded' | 'retried_failed' | 'acknowledged' | 'auto_recovered'
    resolved_at              TEXT NULL                -- RFC3339 when resolution set
    -- 'acknowledged' = founder explicitly read this and accepts the memory
    -- as not-syncable; row stays for audit but no further retries.
);
CREATE INDEX idx_dead_letter_unresolved ON dead_letter(resolved_at) WHERE resolution IS NULL;
```

Refinements vs Shahbaz's opening: replaced binary `acknowledged` with a richer `resolution` enum (pending / retried-success / retried-fail / dropped / auto-recovered), added `payload_format_version` so we can evolve the JSON without a schema migration, added partial index on unresolved entries for fast list queries. `attempts_made` and `failure_reason` self-document the entry (the audit log has the full attempt history, but the row alone is interpretable).

**Trade-offs accepted.** The audit log is authoritative for full attempt history; dead_letter only stores the *last* failure reason. Acceptable: the CLI's `inspect` subcommand cross-references the audit log when needed.

## vault-cli — argument **for** inclusion in T0.1.6

**Recommend: ship `vault-cli` as part of T0.1.6, scoped tightly to dead-letter ops only.**

**Why for (option 1).**
- Founder dogfood (V0.1) is the entire point. A founder hitting a dead-letter on day 2 with no recovery tool means either: (a) `sqlite3` shell with manual SQL — too rough, easy to wreck the audit chain — or (b) wait for T0.1.11 UI which is 5+ tasks away. Both unacceptable for the dogfood loop.
- The CLI is small (≤500 lines). It links to `vault-storage` directly, so no new abstractions: it's a thin clap-based wrapper over the same `StorageBackend` API the Tauri commands will eventually use.
- Building it during T0.1.6 keeps it next to the dead-letter design — when we ship the migration, the CLI tests it. If it's deferred, the next person picking it up has to re-derive the recovery flow.

**Why not (options 2 + 3) rejected.**
- *HTTP debug endpoint on the local MCP server (option 2).* Rejected: adds attack surface to the security-critical MCP layer for an ops tool. Wrong layer.
- *Just logs + manual SQLite (option 3).* Rejected: founder would need to know SQLCipher key material, audit chain seal pattern (BLAKE3 chained), and `payload BLOB` decoding. Operationally hostile.

**Tightness constraint.** New crate `crates/vault-cli/` — binary only. Subcommands V0.1: **`dead-letter list`**, **`dead-letter inspect <id>`**, **`dead-letter retry <id>`**, **`dead-letter acknowledge <id> --reason <text>`** (renamed from `drop` per Shahbaz's terminology — `acknowledge` makes clear the row stays for audit, semantics are "founder read this and accepts loss"). Plus top-level **`divergence-check`** (on-demand entry point from Q3). **Nothing else.** Header comment names what's allowed; PR review rejects anything outside that scope. When V0.2 / V1.0 want a richer admin surface, that's a separate task with its own scope review.

**Authentication (added per Shahbaz clarification).** Every vault-cli invocation requires the **master vault passphrase**. Read in priority order:
1. Env var `VAULT_PASSPHRASE` (preferred for CI / scripted recovery, treated as a secret — never logged).
2. Stdin prompt with no-echo (interactive, for the founder typing it in).

The passphrase is verified by **attempting to open the SQLCipher-encrypted SQLite via `MetadataStore::open(...)`** — if open succeeds, auth passes; the same `SqlCipherKey` derivation as the Tauri app uses (no parallel auth path, no parallel risk surface). If open fails (wrong passphrase or DB file inaccessible), CLI exits with a generic "authentication failed" message — no information leakage about which check failed. Closes the local-malware threat where unauthorised code on the founder's machine could run `vault-cli dead-letter retry --all` without the founder typing the passphrase.

**Trade-offs accepted.** vault-cli is +1 crate in the workspace. ~500–600 lines (slightly larger now with auth + acknowledge — still well under 1k). T0.1.6 ships ~15% bigger than it would otherwise. Worth it for the dogfood-loop unblock + the local-malware mitigation.

## Q5 — Test strategy

**Decision.** Heavy level. Five adversarial tests **simulated via injected fault points, not real OS-level chaos**, plus standard CRUD coverage.

**Why injected faults, not real chaos.** Real `kill -9` in-test, real `tmpfs` constraints, and real corruption-during-write are slow, flaky, and platform-specific. We don't gain confidence that's worth the maintenance cost — the failure-handling code is the same code path whether the trigger is a real `EIO` or an injected one. Use `#[cfg(test)]`-gated fault injection traits on the cascade boundary (a `FaultInjector` parameter on `StorageBackend::new_for_test(...)`); production builds have a no-op injector.

**Adversarial tests (each with property assertions):**
1. **Mid-cascade abort + recovery.** Inject panic between SQLite ack and LanceDB write. Drop the orchestrator (simulates process exit). Construct a new orchestrator on the same files; assert: SQLite has the row, retry queue has the entry, retry runs, all-three-stores converge. Property: every committed memory is eventually present in all three stores OR dead-lettered with audit trail.
2. **Persistent LanceDB write failure → dead-letter.** Inject "always fail" on LanceDB writes. Insert one memory. Assert: 8 retries (timing observed via fast-forwarded clock), then dead-letter row appears with `attempts_made = 8`, CRITICAL log emitted, audit log has 8 entries with `result=Error`.
3a. **Soft LanceDB corruption (silent drop) caught by divergence check.** Insert 10 memories cleanly. Manually delete one row from the LanceDB table (bypass StorageBackend — simulates the realistic production failure where a write *appeared* to succeed but didn't actually commit). Run divergence check. Assert: tier-1 count comparison shows mismatch (10 in SQLite, 9 in LanceDB) → tier-2 sampled-existence catches the missing ID → divergence finding logged + audit event + entry in `pending_sync` for re-enqueue.
3b. **Hard LanceDB corruption (file-format invalid) caught at next open (added per Shahbaz Change 1).** Insert 10 memories. Close the StorageBackend. Manually overwrite a Lance fragment file's first 64 bytes with random bytes (corrupts the Parquet magic header). Open StorageBackend again. Assert: `LanceVectorStore::open` returns `Err`, `StorageBackend::open` propagates this as a CRITICAL log event + audit event tagged `lancedb_corruption_at_open`, and the system enters degraded mode (SQLite still accessible, LanceDB unavailable, vault-cli `dead-letter list` works for triage). The realistic production split — soft corruption (3a) is the common case, hard corruption (3b) is the rare-but-catastrophic case — both must surface to the founder via different paths.
4. **Concurrent cascades on same memory.** Two tokio tasks call `update_memory(id, content_a)` and `update_memory(id, content_b)` concurrently. Per the cascade-ordering invariant in Q1: final state in SQLite is one of `{a, b}` (last-writer-wins by SQLite commit order, T1 < T2 ⇒ B wins), LanceDB and DuckDB converge to that same state via the retry queue's strict-FIFO-per-`memory_id` discipline. Assert: a third reader observing LanceDB at any time during the cascade sees either `a` or `b`, never a torn intermediate; both audit log entries are present in BLAKE3-chain order matching SQLite commit order; `is_fully_synced(id)` returns `true` only after both writes' downstream ops complete.
5. **Retry queue overflow.** Inject "always fail" on LanceDB. Insert 10,001 memories. Assert: SQLite has all 10,001 rows; queue has exactly 10,000 entries; the 10,001st triggers `paused` mode (CRITICAL log + audit event); subsequent inserts succeed in SQLite + go to a `delayed` queue; on injection lifted and queue drained, all 10,001 reach LanceDB.

**Standard CRUD coverage (non-adversarial):** round-trip across the three stores; cascading delete propagates; retry success path (one transient fail → second attempt succeeds, no dead-letter); divergence check on a clean vault returns no findings; vault-cli `dead-letter retry` re-enqueues an entry and the retry succeeds.

**Trade-offs accepted.** Real-OS chaos (true `kill -9`, real `ENOSPC`) is not exercised in CI. Real chaos is reserved for the V0.1.12 founder-dogfood end-to-end task — at that point we have a Tauri app and can crash it under realistic conditions. CI tests the code paths; dogfood tests the OS interactions.

## Background task lifecycle (added per Shahbaz Clarification 4)

**Tasks running in V0.1:**
- **retry worker** (1 task) — drains `retry_queue` FIFO, applies backoff, dead-letters per Q2.
- **divergence detector** (1 task) — runs once on startup + on-demand via vault-cli; NOT a periodic timer in V0.1 (Q3 deferred).
- **dead-letter pruner** — NOT in V0.1; defer to V0.2 when DLQ growth becomes plausible.

**Ownership.** `Application` (T0.1.10) owns lifecycle. T0.1.6 ships a contract:

```rust
impl StorageBackend {
    /// Spawn V0.1 background workers. Caller (Application) owns the
    /// returned handles and is responsible for triggering `cancel` on
    /// shutdown. Workers exit gracefully after their current iteration.
    pub fn spawn_workers(
        &self,
        cancel: tokio_util::sync::CancellationToken,
    ) -> Vec<tokio::task::JoinHandle<()>> { ... }
}
```

T0.1.10 will call this from `Application::start` and trigger the token from `Application::shutdown`.

**Graceful shutdown.** On `cancel.cancel()`, each worker:
1. Finishes its current iteration (e.g., the retry worker completes the entry it's currently processing — does NOT abandon a half-written cascade).
2. Stops polling for new work.
3. Exits cleanly, returning `()` from the spawned future.

**Panic recovery.** Each worker is wrapped in a `panic_recovery_loop`:
- If the worker future panics, log CRITICAL with the panic payload, sleep with backoff (1s → 5s → 30s).
- After 3 restarts, give up: fire CRITICAL audit event `background_worker_giving_up` with task name + last panic, exit the recovery loop. The task is now permanently down for this session.
- A permanently-down retry worker leaves `retry_queue` entries unprocessed; they accumulate; on next app restart the new worker drains them. Founder visibility via vault-cli `dead-letter list` (which shows both DLQ rows AND aged retry_queue entries — small UX add).

**Why not infinite restart.** A worker that panics 3 times in a row has a structural bug — pretending it's transient by restarting forever masks the issue. CRITICAL alert + give-up surfaces the bug to the founder within the same session.
